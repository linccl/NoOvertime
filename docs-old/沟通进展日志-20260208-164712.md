## 2. 关键收敛（记录 001-030）

### 2.1 技术与范围

- 数据库选型确定为 `PostgreSQL`（服务器 Docker 部署）。
- 同步范围确定为“打卡 + 请假（新增/修改/删除）”。
- MVP 鉴权口径确定：配对码 + 恢复码 + Web 只读绑定。

### 2.2 数据库一致性与约束（库层兜底）

- 单写入端语义固化：`writer_device_id + writer_epoch`，并补充旧设备/过期 epoch 提交拒绝策略。
- 同步幂等语义固化：
  - `sync_id` 为“同用户唯一”；
  - 相同 `sync_id + payload_hash` 幂等成功；
  - 相同 `sync_id + 不同 payload_hash` 冲突拒绝。
- 分钟精度固化：入库按分钟截断，`at_utc/local_date/timezone_id/minute_of_day` 一致性约束明确。
- 日/月汇总固化：
  - `day_summaries`、`month_summaries` 增加 `version`；
  - `is_leave_day=true` 时 `adjust_minutes=0`；
  - 请假日 `INCOMPLETE` 的字段组合口径明确。
- 迁移请求与状态机固化：
  - `NORMAL/FORCED` 语义与字段约束固定；
  - 状态不可越级、终态不可流转；
  - 过期请求禁止流转到 `CONFIRMED/COMPLETED`；
  - 同用户同一时刻仅允许一个 `PENDING`；
  - 关键字段（含 `expires_at`）不可变；
  - `NORMAL` 模式下 `from_device_id` 必须等于当前 `writer_device_id`。
- Web 只读绑定固化：
  - `ACTIVE` 会话必须匹配当前 `pairing_code_version`；
  - `REVOKED` 不可逆；
  - 配对码重置触发旧绑定撤销（数据库触发器兜底）。
- `FR-031` 固化：
  - 全天请假日禁止 `source=AUTO` 打卡；
  - 打卡侧 + 请假侧双向触发器兜底；
  - `punch_records.user_id`、`leave_records.user_id` 创建后不可更新，封堵绕过路径。
- 限流固化：
  - 三层维度（细粒度 / 主体全局 / 终端全局）；
  - `window_start` 由数据库时间归一化（UTC 口径）；
  - 成功后不清空失败计数，按窗口自然过期恢复。

### 2.3 复检结论模式

- 多轮复检后结论稳定：除 3 项交付物未落地外，未发现新增结构性阻断。
- 需求、数据库方案、实施计划之间的非阻断文案歧义已收敛（字段必填性、请假与同步口径、汇总字段口径）。

## 3. 三项交付物执行计划加固（记录 031-063）

### 3.1 冻结目标（不扩边界）

- 仅落地 3 项交付物：`001_init.sql`、`API契约草案.md`、`数据库验证记录.md`。
- 不扩展 Android/Web 业务实现。
- 要求形成“计划 -> 实施 -> 验证 -> 沟通留痕”的可审计闭环。

### 3.2 覆盖范围与映射

- FR 覆盖明确纳入：`FR-008/016/021/022/023/026/027/028/029/031/032/033`。
- 固定两张映射表为必交付内容：
  - `FR -> 接口覆盖映射`
  - `数据库拒绝原因 -> API 错误码映射`（含 `REJECTED/NOOP` 边界）

### 3.3 执行与验真硬门槛（最终口径）

- 前置检查：
  - `SHOW server_version_num` 必须 `>= 160000` 且 `< 170000`；
  - `pgcrypto` 必须“已安装或可创建”；
  - 扩展白名单固定：仅允许 `plpgsql`（系统默认）与 `pgcrypto` 预装；
  - 非系统 `schema` 白名单固定：仅允许 `public` 与允许扩展（`plpgsql/pgcrypto`）的 `extnamespace`；
  - 空库首次执行判定：排除系统对象与扩展归属对象后，在允许非系统 `schema` 集合内，非扩展归属的 `enum/table/view/materialized view/sequence/index/constraint/function/procedure/trigger` 数量为 `0`。
- 迁移脚本执行：
  - `001_init.sql` 必须 `BEGIN ... COMMIT` 单事务；
  - 执行命令固定 `psql -v ON_ERROR_STOP=1`；
  - 任一 DDL 失败即整批回滚，不留半成功对象。
- 对象完整性核对：
  - 覆盖 `extension/enum/table/index/constraint/function/trigger`；
  - 核对键固定为 `schema + 对象类型 + 对象标识`；
  - 细则：
    - `function`：函数名 + 参数签名；
    - `trigger`：所属表 + 触发器名；
    - `constraint/index`：`schema + table_name + object_name`。
- NOOP/冲突证据双层化：
  - DB 层：`INSERT/UPDATE/DELETE` 三类计数分别为 `0`，且限定同 `test_user_id` + 同用例快照窗口；
  - 统计表固定：`punch_records`、`leave_records`、`day_summaries`、`month_summaries`、`sync_commits`；
  - API 契约层：给出 `gate_result(APPLIED/NOOP/REJECTED)`、`gate_reason` 与规则映射。
- 失败用例标准化：
  - DB 层必须记录 `SQLSTATE`；
  - API 层必须记录 `error_code`；
  - 必须记录“预期错误标识 vs 实际错误标识”一致性比对。
- 指纹与留痕：
  - 输入基线文档：执行前/后 `sha256` 必须一致；
  - `001_init.sql` 与 `API契约草案.md`：执行前/后 `sha256` 必须留痕（首次创建允许执行前 `ABSENT`）；
  - `数据库验证记录.md`：执行后最终 `sha256` 与变更原因回写沟通日志（避免同文件自引用）。
- 时间与时区证据：
  - 执行语义固定：先 `UTC` 执行与校验，再切 `Asia/Hong_Kong` 仅做对照留痕；
  - 关键动作必须写“动作完成时间 + 记录时间”，且时间差 `<= 5 分钟`。
- 无服务实现边界：
  - 不得标记“联调通过”；
  - `FR-029/FR-033` 在本轮允许按 `API契约` 层证据验真，联调结果标记为后续待执行。
