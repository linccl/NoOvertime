## 2. 分阶段关键收敛

### 2.1 阶段 A（记录 001-030）：基础技术口径收敛

- 技术选型与范围：
  - 数据库为 `PostgreSQL`（Docker 环境）。
  - 同步范围固定为打卡与请假（新增/修改/删除）。
  - MVP 鉴权固定为“配对码 + 恢复码 + Web 只读绑定”。
- 数据一致性与幂等：
  - 单写入端语义固定为 `writer_device_id + writer_epoch`。
  - `sync_id` 固定为同用户唯一；`sync_id + payload_hash` 同则幂等成功，不同则冲突拒绝。
  - 分钟精度、日/月汇总 `version`、请假日规则与状态机约束已固化。
- 关键规则：
  - `FR-031` 固化：全天请假日禁止 `source=AUTO` 打卡，触发器双向兜底，`user_id` 创建后不可更新。
  - 限流固定为三层维度，`window_start` 使用数据库 UTC 时间归一化。

## 3. 最终有效规则与验收门槛（当前执行口径）

- 基线优先级：
  - 规则依据按 `docs/需求文档.md` -> `docs/数据库方案草案.md` -> `docs/实施计划.md` -> `docs/三项交付物执行计划.md (Frozen v1.60)`。
  - `docs/沟通进展日志.md` 用于审计留痕，不作为可漂移规则源。
- 前置检查门槛：
  - `SHOW server_version_num` 必须 `>= 160000` 且 `< 170000`。
  - `pgcrypto` 必须“已安装或可创建”（事务探测失败需保留 `SQLSTATE`）。
  - 扩展基线：`plpgsql`（系统默认）必须存在，`pgcrypto` 必须“已安装或可创建”；其他预装扩展允许存在，但需记录扩展名与 `extnamespace`，并在空库判定中按“扩展归属对象”排除统计。
  - 非系统 `schema` 白名单为 `public + 允许扩展的 extnamespace`。
  - 空库首次执行判定：在允许集合内，对非扩展归属对象计数为 `0`，对象类型覆盖 `enum/table/view/materialized view/sequence/index/constraint/function/procedure/trigger`。
- 迁移执行门槛：
  - `db/migrations/001_init.sql` 必须 `BEGIN ... COMMIT` 单事务执行。
  - 执行命令固定 `psql -X -v ON_ERROR_STOP=1`。
  - 当 `pgcrypto` 位于非 `public` `schema` 时，需使用临时 `search_path=public,<pgcrypto_ext_schema>` 并留痕。
  - 任一 DDL 失败即整体回滚，不允许半成功对象残留。
- 对象完整性门槛：
  - 采用“数据库方案快照 + `001_init.sql` 清单”双基线交叉核对，按核心对象域（A ∪ B）差异判定失败。
  - 基线 A 提取方法固定为“数据库方案第 3/4/7/8 章 + `object_type/schema/object_identifier/source_anchor` 字段模板”（`extension` 仅取第 8 章 `CREATE EXTENSION`）；未显式命名约束按 `UNNAMED_CONSTRAINT::<table_name>::<source_anchor>` 规范化并记录 A/B 映射。
  - 核对键固定为：`extension` 使用 `对象类型 + extension_name`（`extnamespace` 仅留痕）；其余对象使用 `schema + 对象类型 + 对象标识`（`function` 含参数签名，`trigger` 含所属表）。
- 门控与失败证据门槛：
  - API 层必须输出 `gate_result(APPLIED/NOOP/REJECTED)`、`gate_reason` 与规则映射。
  - `NOOP` 证据必须按门控类型记录 DB 层计数，并限定同 `test_user_id` 与同快照窗口，统计表固定为 `punch_records`、`leave_records`、`day_summaries`、`month_summaries`、`sync_commits`：`REPLAY_NOOP/SYNC_ID_CONFLICT` 为 5 张表 `INSERT/UPDATE/DELETE=0`；`LOW_OR_EQUAL_VERSION`（新 `sync_id`）为 4 张业务表 `INSERT/UPDATE/DELETE=0` 且 `sync_commits INSERT=1, UPDATE=0, DELETE=0`。
  - `sync_id` 门控副作用矩阵固定为 `REPLAY_NOOP / SYNC_ID_CONFLICT / LOW_OR_EQUAL_VERSION`。
  - 写入生效门控固定为 `APPLIED_WRITE -> APPLIED`，并要求“同一 `test_user_id`、同一用例窗口”的落库变化证据。
  - 失败用例必须保留分层标识：`P0001` 场景为 `SQLSTATE + error_key`；`23502` 场景为 `SQLSTATE + table_name + column_name`；其余约束拒绝场景为 `SQLSTATE + constraint_name`，并记录“预期 vs 实际”一致性与唯一规则映射。
- 指纹与审计门槛：
  - 执行前后 `sha256` 留痕为强制项（首次创建可记 `ABSENT`）。
  - `docs/数据库验证记录.md` 最终 hash 与变更原因必须外置回写本日志，避免自引用。
  - 执行语义固定为先 `UTC` 校验，再切换 `Asia/Hong_Kong` 仅作对照留痕。
  - 时间证据必须写“动作完成时间 + 记录时间”，差值 `<= 5 分钟`；超时必须解释原因，禁止伪造时间。
  - 复审闭环必须记录 `review_checklist_id` 与 `evidence_anchor_set（>=3）`，连续轮次不得重复。
- 范围边界门槛：
  - 本轮不标记“联调通过”。
  - `FR-029/FR-033` 允许以 API 契约证据验真，联调结果作为后续事项。
