## 4. 冻结执行步骤

1. 产出初始化迁移脚本 `001_init.sql`
- 执行顺序固定：`extension -> enum -> table -> index -> function -> trigger`
- 覆盖数据库方案中的关键约束与触发器（打卡成对、迁移状态机、Web 绑定版本、单写入端兜底、限流窗口归一化、FR-031 兜底、记录归属不可变）
- 脚本定位为“空库初始化一次性脚本”，不内置测试种子数据
- 事务边界固定：
  - `001_init.sql` 必须在单事务内执行（`BEGIN ... COMMIT`），并要求执行端启用 `ON_ERROR_STOP`
  - 任意 DDL 失败必须触发整批回滚，不允许出现“部分对象已落库”的半成功状态
- 对象命名规范固定：
  - 所有需验真的约束/索引/函数/触发器必须显式命名（禁止依赖数据库默认命名）
  - 核心对象清单与 `pg_catalog` 对比时，匹配键按对象类型固定：`extension` 使用 `对象类型 + extension_name`（`extnamespace` 仅留痕，不作为全等硬匹配键）；其余对象使用“`schema + 对象类型 + 对象标识`”（`function` 使用 `schema + function_name + argument_types`；`trigger` 使用 `schema + table_name + trigger_name`；`constraint` 使用 `schema + table_name + constraint_name`；`index` 使用 `schema + table_name + index_name`）
- 失败标识规范固定：
  - 失败标识采用“`SQLSTATE + 规则标识`”双标识分层口径
  - 触发器/函数显式 `RAISE` 的业务拒绝：`SQLSTATE` 固定为 `P0001`，并要求 `error_key` 前缀（`[error_key=<KEY>] ...`）
  - 数据完整性约束拒绝分层映射：`UNIQUE/CHECK/FK` 等约束拒绝保留数据库原生 `SQLSTATE`（如 `23505/23514/23503`），并通过 `SQLSTATE + constraint_name` 映射到唯一 `error_key`；`NOT NULL`（`23502`）拒绝使用 `SQLSTATE + table_name + column_name` 映射到唯一 `error_key`
  - 同一业务规则必须稳定映射到同一 `error_key`（禁止同义多 key）
- 执行命令模板固定：
  - 建库与校验统一使用 `psql -X -v ON_ERROR_STOP=1`（禁止读取 `.psqlrc`，避免本地初始化脚本污染执行语义）
  - 对象核对阶段固定 `SET search_path = public;`，禁止依赖客户端默认 `search_path`
  - 若 `pgcrypto` 已安装且 `extnamespace <> 'public'`，执行 `001_init.sql` 前必须临时设置 `SET search_path = public, <pgcrypto_ext_schema>;` 并在验证记录留痕，避免 `gen_random_uuid()` 解析失败
  - 会话先固定 `SET TIME ZONE 'UTC';` 执行建库与 UTC 归一化相关校验；完成后再切换 `SET TIME ZONE 'Asia/Hong_Kong';` 仅补采日志对照时间戳（不重复执行迁移）
- 执行前置检查固定：
  - 适用范围：仅用于“空库首次执行 `001_init.sql`”前；不用于“同库二次执行失败验证”场景
  - PostgreSQL 主版本为 16（以 `SHOW server_version_num` 校验：`>= 160000` 且 `< 170000`）
  - `pgcrypto` 已安装或可创建（硬门槛；若两者均不满足，本轮执行立即中止，并在验证记录中明确失败原因、阻塞项与申请提权路径）
  - `pgcrypto` “可创建”判定必须使用事务探测 SQL：`BEGIN; CREATE EXTENSION IF NOT EXISTS pgcrypto; ROLLBACK;`；若探测失败，必须记录 `SQLSTATE + 错误信息`
  - 若探测结论为“已安装且 extnamespace 非 public”，则保持既有扩展归属 `schema`，仅通过会话 `search_path` 兼容函数解析，禁止在本轮迁移中变更扩展归属 `schema`
  - 扩展基线固定：必须存在 `plpgsql`（系统默认），且 `pgcrypto` 已安装或可创建；其他预装扩展允许存在，但必须在验证记录中留痕（扩展名 + `extnamespace`），并在空库判定中按“扩展归属对象”排除统计
  - 非系统 `schema` 基线固定：必须至少包含 `public`；其他非系统 `schema` 允许存在，但若其内存在“非扩展归属对象”，直接判定为脏库并中止本轮执行
  - 目标库为空（判定口径固定：仅排除系统对象与“扩展归属对象”；在“允许非系统 `schema` 集合”内，非扩展归属的 `enum/table/view/materialized view/sequence/index/constraint/function/procedure/trigger` 总数必须为 0；扩展归属判定以 `pg_depend.deptype = 'e'` 为准），避免一次性初始化脚本在脏库执行
  - 空库判定 SQL 与统计结果必须写入验证记录，作为前置检查留痕证据
  - 任一前置检查失败，不得继续建库与验证步骤

2. 产出 API 契约草案
- 覆盖接口：
  - 同步上报（Punch/LeaveRecord/DaySummary/MonthSummary 原子提交）
  - 迁移申请/确认/强制接管
  - 配对码重置
  - 恢复码重置
  - Web 只读绑定创建与鉴权
- 每个接口必须包含：
  - 请求/响应
  - 错误码
  - 幂等字段（如不适用需显式标注 `N/A`）
  - 鉴权要求
  - 限流策略
- API 契约结构固定补充：
  - FR -> 接口覆盖映射表（逐条映射 `FR-008/FR-016/FR-021/FR-022/FR-023/FR-026/FR-027/FR-028/FR-029/FR-031/FR-032/FR-033`）
  - 数据库拒绝原因 -> API 错误码映射表（含拒绝与 NOOP 丢弃语义边界）
  - 门控结果字段规范：涉及幂等/版本门控的接口响应必须显式返回 `gate_result`（`APPLIED/NOOP/REJECTED`）与 `gate_reason`（如 `LOW_OR_EQUAL_VERSION`/`SYNC_ID_CONFLICT`）
  - 门控字段字典固定：`gate_result` 与 `gate_reason` 必须在契约中定义为闭集枚举；每个 `gate_reason` 必须映射到唯一数据库规则（`error_key` 或门控规则 ID），禁止出现未映射值
  - 门控返回矩阵固定：相同 `sync_id + 相同 payload_hash` 返回 `gate_result=NOOP`、`gate_reason=REPLAY_NOOP`（语义为幂等成功，不重复生效）；相同 `sync_id + 不同 payload_hash` 返回 `gate_result=REJECTED`、`gate_reason=SYNC_ID_CONFLICT`；低/同版本 + 新 `sync_id` 返回 `gate_result=NOOP`、`gate_reason=LOW_OR_EQUAL_VERSION`；写入实际生效返回 `gate_result=APPLIED`、`gate_reason=APPLIED_WRITE`
  - `payload_hash` 生成口径固定（用于 `sync_id` 幂等判定，避免同语义异哈希）：
    - 哈希算法固定为 `SHA-256`，输出固定为小写十六进制
    - 哈希输入固定为 UTF-8 编码“规范化 JSON”字符串（无额外空白）
    - 规范化规则固定：顶层键顺序 `user_id/device_id/writer_epoch/sync_id/punch_records/leave_records/day_summaries/month_summaries`；对象字段按契约字段顺序输出；数组按稳定业务主键排序（`id` 或日期键）；时间字段统一为 UTC ISO8601 分钟粒度（`YYYY-MM-DDTHH:mm:00Z`）
    - 同一业务语义在字段顺序或空白差异下必须得到同一 `payload_hash`；业务语义发生变化时必须得到不同 `payload_hash`
- 错误码/门控映射固定覆盖：
  - 限流阻断
  - 幂等冲突
  - 状态机越级
  - 普通迁移来源设备不匹配
  - 迁移不可变字段更新（含 `expires_at` 续期）
  - Web 撤销不可逆
  - 配对码重置后旧配对码绑定失败
  - 恢复码重置后旧恢复码验证失败
  - FR-008（缺 `START` 直接写 `END`）
  - `stale/non-writer device commit`
  - 封禁窗口内拒绝
  - 低/同版本提交（含新 `sync_id`）NOOP 丢弃
- 时间精度固定：
  - API 入库前按分钟向下取整（截断）
  - `local_date/timezone_id/minute_of_day` 与截断后时间戳不一致时返回参数错误
- 恢复码语义固定：
  - 恢复码重置后旧恢复码立即失效
  - 新恢复码明文仅在“生成/重置响应”返回一次，后续仅允许哈希校验路径
  - 恢复码校验固定不区分大小写：请求入参必须先做大写归一（A-Z、0-9）再参与哈希校验；同值大小写变体视为同一恢复码
- 配对码语义固定：
  - 配对码格式固定为 8 位数字（`^[0-9]{8}$`）；不符合格式时返回参数错误，禁止进入绑定/迁移流程

3. 产出数据库验证记录
- 建库与回滚：
  - 空库执行通过
  - 同库二次执行按预期失败且可定位
  - 回滚演练可恢复到干净状态
  - 核心对象完整性校验通过（扩展/枚举/表/索引/约束/函数/触发器与 `001_init.sql` 设计清单一致）
  - 验证记录脱敏输出（配对码/恢复码/token 不写明文）
- 对象完整性校验基线固定：
  - 采用“双基线交叉核对”：基线 A 为 `docs/数据库方案草案.md` 的对象要求快照，基线 B 为 `001_init.sql` 的核心对象清单
  - 基线 A 生成方法固定：从 `docs/数据库方案草案.md` 第 3/4/7/8 章提取对象（`extension` 仅从第 8 章 `CREATE EXTENSION` 语句提取，其余对象从第 3/4/7 章提取），快照字段固定为 `object_type/schema/object_identifier/source_anchor`
  - 基线 A 提取过滤规则固定：仅统计对象定义 DDL（`CREATE TYPE`、`CREATE TABLE`、`CREATE [UNIQUE] INDEX`、`ALTER TABLE ... ADD CONSTRAINT`、`CREATE OR REPLACE FUNCTION`、`CREATE [CONSTRAINT] TRIGGER`、`CREATE EXTENSION`）；必须排除“示例/建议/模板”代码块与非对象语句（如第 7.9 节建议模板中的 `RAISE` 片段），禁止将说明性 SQL 片段计入对象快照
  - 基线 A 的 `source_anchor` 固定为 `文件路径:行号`；验证记录必须留存“提取命令 + 人工复核说明”。人工复核默认仅允许补漏；若命中“数据库方案未显式命名的约束（如 CHECK）”，允许按 `UNNAMED_CONSTRAINT::<table_name>::<source_anchor>` 补充标准化标识，并在复核说明中记录其与基线 B 显式约束名的映射关系
  - 两份基线都必须逐项标注“对象类型 + 对象标识”（`extension/enum/table/index/constraint/function/trigger`；`extension` 的 `object_identifier` 固定为 `extension_name`，`extnamespace` 作为附加留痕字段；`function` 记录签名，`trigger/constraint/index` 记录所属表；基线 A 中未命名约束使用标准化标识，基线 B 使用显式约束名）
  - 两份基线中的核心业务对象 `schema` 必须显式为 `public`（`extension` 除外）
  - 差异判定范围固定为“核心对象域（A ∪ B）”；`pg_catalog` 比对仅对核心对象域做全等校验
  - 核心对象域外对象（含其他预装扩展对象）只留痕不判失败；但若命中前置检查定义的“非扩展归属对象”违规，仍判失败
  - 验证记录必须同时给出：基线 A 快照、基线 B 快照、`pg_catalog` 实际查询结果、A/B 各自差异对比结论；核心对象域内任一差异均判失败
- 验证记录全局元信息固定：
  - 必须记录：`PostgreSQL 版本`、`server TimeZone`、`session TimeZone`（UTC 与 `Asia/Hong_Kong` 两次会话快照）、`database 名称`、`执行开始时间（+08:00）`、`执行结束时间（+08:00）`
  - 涉及 UTC 归一化或时间窗口断言的用例，必须同时记录 `UTC` 与 `+08:00` 两套时间戳
  - 必须记录本轮交付物指纹：`db/migrations/001_init.sql` 的 `sha256`、`docs/API契约草案.md` 的 `sha256`
  - 输入基线文档（`docs/需求文档.md`、`docs/数据库方案草案.md`、`docs/实施计划.md`）的指纹留痕必须包含“执行前 hash”与“执行后 hash”，且二者一致；不一致则判定本轮验证无效
  - 规则基线文档（`docs/三项交付物执行计划.md`）的指纹留痕必须包含“执行前 hash”与“执行后 hash”，且二者一致；不一致则判定为“执行中基线漂移”，本轮验证无效并需按新版本重跑
  - 本轮产出文件中，`db/migrations/001_init.sql` 与 `docs/API契约草案.md` 的指纹留痕必须包含“执行前 hash”与“执行后 hash”；执行前不存在时固定标记为 `ABSENT`，执行后必须记录实际 `sha256`
  - `docs/数据库验证记录.md` 仅在验证记录中记录“执行前 hash”（首次创建时为 `ABSENT`）；执行后最终版 `sha256` 与变更原因统一回写 `docs/沟通进展日志.md`，避免同文件自引用
  - 若 `db/migrations/001_init.sql` 或 `docs/API契约草案.md` 存在“执行前/执行后 hash 不一致”，必须在验证记录中附“变更原因 + 对应步骤”说明
- 正反例验证必须覆盖：
  - 原子同步：`Punch/LeaveRecord/DaySummary/MonthSummary/sync_commits` 同事务提交（失败回滚不留部分成功）
  - 普通迁移成功与强制迁移成功
  - 接管原子切换：迁移完成后 `writer_device_id` 切换、`writer_epoch` 递增、旧写入设备状态变更为 `REVOKED`
  - 普通迁移越级完成拒绝
  - 普通迁移 `from_device_id = 当前 writer_device_id` 成功路径
  - 普通迁移 `from_device_id != 当前 writer_device_id` 拒绝路径
  - 迁移请求关键字段更新拒绝（`mode/from_device_id/to_device_id/recovery_code_verified`）
  - 迁移请求 `expires_at` 续期更新拒绝
  - 过期迁移请求流转到 `CONFIRMED/COMPLETED` 拒绝
  - 存在过期 `PENDING` 请求时，新迁移申请可成功并自动将旧请求转为 `EXPIRED`
  - 配对码重置撤销旧绑定
  - Web 撤销会话不可逆与版本一致性
  - `sync_id` 幂等：相同 `sync_id + 相同 payload_hash` 语义为幂等成功，`API契约` 层返回 `gate_result=NOOP`、`gate_reason=REPLAY_NOOP`
  - `sync_id` 冲突：相同 `sync_id + 不同 payload_hash` 被拒绝，`API契约` 层返回 `gate_result=REJECTED`、`gate_reason=SYNC_ID_CONFLICT`
  - `payload_hash` 规范一致性：同一业务语义载荷在字段顺序/空白差异下归一化后哈希一致
  - `payload_hash` 差异性：业务语义发生变化时哈希变化，并触发 `SYNC_ID_CONFLICT`
  - `sync_id` 重放无副作用：相同 `sync_id + 相同 payload_hash` 重试时，业务表与 `sync_commits` 均不新增/改写
  - `sync_id` 冲突无副作用：相同 `sync_id + 不同 payload_hash` 被拒绝且业务表与 `sync_commits` 均不新增/改写
  - 低/同版本 + 新 `sync_id` 门控：业务表不新增/改写，`sync_commits` 新增 1 行；`API契约` 层返回 `gate_result=NOOP`、`gate_reason=LOW_OR_EQUAL_VERSION`（`DB` 层以门控规则映射与副作用计数验真）
  - 单写入端兜底（非当前写入端或过期 `writer_epoch` 拒绝）
  - 版本门控（低版本不覆盖、同版本按 `NOOP` 不覆盖、高版本可覆盖）
  - FR-008（缺 `START` 直接写 `END` 拒绝）
  - FR-031（`FULL_DAY` 与 `AUTO` 不可共存）
  - FR-032（分钟截断与数据库分钟约束）
  - FR-032 字段一致性：`local_date/timezone_id/minute_of_day` 与时间戳不一致写入拒绝
  - FR-021/FR-022：当前配对码可绑定 Web；重置后旧配对码绑定失败且新配对码绑定成功；非 8 位数字配对码请求被拒绝

## 7. 变更控制

- 本文档冻结后，不得在实施过程中临时扩展范围。
- 如需调整阈值、规则或验收口径，必须先更新对应上游规则文档（`需求文档/数据库方案草案/实施计划`）与本文，并在沟通日志记录变更原因与影响，再执行实现。
- 本文每次 `Frozen` 升级必须同步刷新“上游基线快照（3 份上游文档 `baseline_sha256`）”并留痕；未刷新前不得进入实施、验证与复审计数。
- 执行中若触发上游基线漂移，必须先记录“漂移证据 + 停执行结论”，再进入文档同步与复审重计流程，禁止带漂移状态继续验证。
- `docs/沟通进展日志.md` 的历史条目仅用于审计追溯；若与本文存在表述差异，以本文当前 Frozen 版本为执行与验收准绳。
