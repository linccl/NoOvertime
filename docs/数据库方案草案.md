# NoOvertime PostgreSQL MVP 草案

> 状态：Draft v0.7  
> 最后更新：2026-02-07  
> 适用范围：Android App（唯一写入端）+ Web 只读看板

## 1. 选型结论

- 数据库：PostgreSQL 18.1（Docker 部署）
- 原因：
  - 需要强约束：同日同类型打卡唯一、单写入设备、迁移接管原子切换
  - 需要幂等同步：`UPSERT` + 版本号/更新时间控制
  - 需要可维护的规则承载：`CHECK`、`ENUM`、部分唯一索引（`partial unique index`）

## 2. 数据设计原则

- 时间最小粒度为分钟：客户端与 API 接入层在入库前统一截断到分钟；数据库入库层通过约束保证最终仅落 `秒=0` 数据。
- 软删除优先：删除打卡用 `deleted_at`，用于同步与重算追踪。
- Web 只读：Web 仅读取 `day_summaries`、`month_summaries`，不做计算。
- 移动端主导计算：服务端主要负责存储、一致性约束、鉴权与限流。
- 同步原子性：一次同步内的 Punch/LeaveRecord/DaySummary/MonthSummary 与同步提交元数据同事务提交，避免 Web 与移动端短暂口径不一致。

## 4. 核心表结构（MVP）

### 4.1 用户与设备

```sql
CREATE TABLE users (
  user_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  pairing_code VARCHAR(8) NOT NULL UNIQUE,
  pairing_code_version BIGINT NOT NULL DEFAULT 1,
  pairing_code_updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  recovery_code_hash TEXT NOT NULL,
  writer_device_id UUID,
  writer_epoch BIGINT NOT NULL DEFAULT 1,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (pairing_code ~ '^[0-9]{8}$')
);

CREATE TABLE devices (
  device_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
  device_name TEXT,
  status device_status NOT NULL DEFAULT 'ACTIVE',
  revoked_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (user_id, device_id)
);

CREATE INDEX idx_devices_user_status ON devices(user_id, status);

-- 由于 users 与 devices 相互依赖，writer_device 约束在两表创建后追加
ALTER TABLE users
ADD CONSTRAINT fk_users_writer_device
FOREIGN KEY (user_id, writer_device_id)
REFERENCES devices(user_id, device_id)
DEFERRABLE INITIALLY DEFERRED;
```

### 4.2 打卡记录

```sql
CREATE TABLE punch_records (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
  local_date DATE NOT NULL,
  type punch_type NOT NULL,
  at_utc TIMESTAMPTZ NOT NULL,
  timezone_id TEXT NOT NULL,
  minute_of_day SMALLINT NOT NULL,
  source punch_source NOT NULL,
  deleted_at TIMESTAMPTZ,
  version BIGINT NOT NULL DEFAULT 1,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (minute_of_day BETWEEN 0 AND 1439),
  CHECK (at_utc = date_trunc('minute', at_utc)),
  CHECK (local_date = (at_utc AT TIME ZONE timezone_id)::date),
  CHECK (
    minute_of_day =
    (
      EXTRACT(HOUR FROM (at_utc AT TIME ZONE timezone_id))::INT * 60
      + EXTRACT(MINUTE FROM (at_utc AT TIME ZONE timezone_id))::INT
    )
  )
);

-- 仅未删除记录参与唯一性：同用户同日同类型最多一条
CREATE UNIQUE INDEX uk_punch_active_unique
ON punch_records(user_id, local_date, type)
WHERE deleted_at IS NULL;

CREATE INDEX idx_punch_user_date ON punch_records(user_id, local_date);
CREATE INDEX idx_punch_user_updated ON punch_records(user_id, updated_at DESC);
```

### 4.4 日汇总与月汇总（Web 只读来源）

```sql
CREATE TABLE day_summaries (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
  local_date DATE NOT NULL,
  start_at_utc TIMESTAMPTZ,
  end_at_utc TIMESTAMPTZ,
  is_leave_day BOOLEAN NOT NULL DEFAULT FALSE,
  leave_type leave_type,
  is_late BOOLEAN,
  work_minutes INTEGER,
  adjust_minutes INTEGER,
  status summary_status NOT NULL DEFAULT 'INCOMPLETE',
  version BIGINT NOT NULL DEFAULT 1,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (work_minutes IS NULL OR work_minutes >= 0),
  CHECK (adjust_minutes IS NULL OR adjust_minutes % 30 = 0),
  CHECK (is_leave_day = FALSE OR adjust_minutes = 0),
  CHECK (
    (is_leave_day = TRUE AND leave_type IS NOT NULL)
    OR (is_leave_day = FALSE AND leave_type IS NULL)
  ),
  CHECK (
    (
      status = 'INCOMPLETE'
      AND work_minutes IS NULL
      AND (
        (is_leave_day = TRUE AND adjust_minutes = 0)
        OR (is_leave_day = FALSE AND adjust_minutes IS NULL)
      )
    )
    OR (status = 'COMPUTED' AND work_minutes IS NOT NULL AND adjust_minutes IS NOT NULL)
  )
);

CREATE UNIQUE INDEX uk_day_summary_user_date
ON day_summaries(user_id, local_date);

CREATE INDEX idx_day_summary_user_date ON day_summaries(user_id, local_date DESC);

CREATE TABLE month_summaries (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
  month_start DATE NOT NULL,
  work_minutes_total INTEGER NOT NULL DEFAULT 0,
  adjust_minutes_balance INTEGER NOT NULL DEFAULT 0,
  version BIGINT NOT NULL DEFAULT 1,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (date_trunc('month', month_start)::date = month_start)
);

CREATE UNIQUE INDEX uk_month_summary_user_month
ON month_summaries(user_id, month_start);

CREATE INDEX idx_month_summary_user_month
ON month_summaries(user_id, month_start DESC);
```

### 4.5 迁移与接管事件

```sql
CREATE TABLE migration_requests (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
  from_device_id UUID,
  to_device_id UUID NOT NULL,
  mode migration_mode NOT NULL,
  status migration_status NOT NULL DEFAULT 'PENDING',
  recovery_code_verified BOOLEAN NOT NULL DEFAULT FALSE,
  expires_at TIMESTAMPTZ NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (from_device_id IS NULL OR from_device_id <> to_device_id),
  CHECK (expires_at > created_at),
  CHECK (
    (mode = 'NORMAL' AND from_device_id IS NOT NULL AND recovery_code_verified = FALSE)
    OR (mode = 'FORCED' AND recovery_code_verified = TRUE)
  ),
  FOREIGN KEY (user_id, from_device_id) REFERENCES devices(user_id, device_id),
  FOREIGN KEY (user_id, to_device_id) REFERENCES devices(user_id, device_id)
);

CREATE INDEX idx_migration_user_status
ON migration_requests(user_id, status, created_at DESC);

-- 同一用户同一时刻只允许一个待确认迁移
CREATE UNIQUE INDEX uk_migration_user_pending
ON migration_requests(user_id)
WHERE status = 'PENDING';
```

### 4.6 配对/恢复码安全限流（数据库承载）

```sql
CREATE TABLE security_attempt_windows (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  scene security_scene NOT NULL,
  subject_hash TEXT NOT NULL,
  client_fingerprint_hash TEXT NOT NULL,
  window_start TIMESTAMPTZ NOT NULL,
  fail_count INTEGER NOT NULL DEFAULT 0,
  blocked_until TIMESTAMPTZ,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (fail_count >= 0),
  UNIQUE (scene, subject_hash, client_fingerprint_hash, window_start)
);

-- window_start 写入前统一做窗口归一化（UTC）：
-- 10 分钟窗口场景：对齐到整 10 分钟边界
-- 24 小时窗口场景：对齐到当日 00:00:00
-- 通过触发器 `trg_normalize_security_window_start` 在数据库层强制执行。

CREATE INDEX idx_security_blocked_until
ON security_attempt_windows(scene, blocked_until);
```

### 4.7 同步原子批次与幂等提交

```sql
CREATE TABLE sync_commits (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
  device_id UUID NOT NULL,
  writer_epoch BIGINT NOT NULL,
  sync_id UUID NOT NULL,
  payload_hash TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'APPLIED',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  applied_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (status IN ('APPLIED', 'REJECTED')),
  FOREIGN KEY (user_id, device_id) REFERENCES devices(user_id, device_id),

CREATE INDEX idx_sync_commits_user_created
ON sync_commits(user_id, created_at DESC);
```

```sql
CREATE TABLE web_read_bindings (
```

## 5. 关键一致性策略

- 单写入端：写请求必须带 `device_id` 与 `writer_epoch`，服务端校验：
  - `device_id == users.writer_device_id`
  - `writer_epoch == users.writer_epoch`
- 数据库兜底：`sync_commits` 写入前通过触发器二次校验 `device_id/writer_epoch` 与 `users` 当前写入端一致，防止服务层漏校验。
- 单写入端归属：`users(user_id, writer_device_id)` 通过复合外键强制约束到本用户设备。
- 换机接管（普通）：旧机确认后事务内执行：
  - 更新 `users.writer_device_id = 新机`
  - `users.writer_epoch = users.writer_epoch + 1`
  - 旧机 `devices.status = REVOKED`
- 强制接管：校验 `pairing_code + recovery_code_hash` 成功后，执行同样切换事务。
- 迁移归属：`migration_requests` 中 `from/to` 设备都必须属于当前 `user_id`。
- 迁移模式约束：`NORMAL` 必须带 `from_device_id` 且 `recovery_code_verified=FALSE`；`FORCED` 必须 `recovery_code_verified=TRUE`。
- 普通迁移来源设备校验：`NORMAL` 模式下 `from_device_id` 必须等于 `users.writer_device_id`（当前写入端）。
- 迁移请求不可篡改：创建后 `mode/from_device_id/to_device_id/recovery_code_verified/expires_at` 不允许更新，防止通过改模式或续期绕过确认流程。
- 新迁移申请自动清理：插入新请求前先将同用户已过期 `PENDING` 请求转为 `EXPIRED`，避免历史脏状态阻塞新申请。
- 全天请假禁自动打卡：通过数据库约束触发器确保 `FULL_DAY` 请假日与 `source=AUTO` 打卡不可共存，兜底 FR-031。
- 打卡/请假归属不可变：`punch_records.user_id` 与 `leave_records.user_id` 创建后不可更新，防止跨用户改写绕过业务约束。
- Web 只读绑定：Web 端校验配对码后创建 `web_read_bindings` 会话；配对码重置时通过数据库触发器自动撤销旧会话，避免遗漏。
- Web 绑定版本：`ACTIVE` 会话必须匹配当前 `users.pairing_code_version`，防止旧版本绑定被误激活。
- Web 绑定状态不可逆：`REVOKED` 会话禁止恢复为 `ACTIVE`。

## 6. 幂等与原子同步规则

- MVP 固定策略：所有可同步实体统一使用 `version` 字段；客户端负责单调递增提交版本。
- 服务端 `UPSERT` 策略：
  - `EXCLUDED.version > target.version` 时覆盖
  - 否则丢弃为重复/旧数据
- 低版本或同版本重放提交按 NOOP 处理：不报错、不覆盖已落库更高（或相同）版本数据。
- 删除采用软删除并同步 `deleted_at`，避免多端误复活。
- `sync_id` 语义（基于 `sync_commits`）：
  - 同一 `user_id` 下，相同 `sync_id + payload_hash`：视为重试，返回幂等成功，不重复生效
  - 同一 `user_id` 下，相同 `sync_id + 不同 payload_hash`：返回冲突错误并拒绝
- 一次同步请求必须在同一事务中提交：
  - `punch_records`
  - `leave_records`
  - `day_summaries`
  - `month_summaries`
  - `sync_commits`
- 事务成功后才返回成功，确保 Web 端读取到的统计与移动端最新同步点一致。

### 6.1 限流阈值基线（MVP）

- `WEB_PAIR_BIND`：5 次 / 10 分钟后封禁 30 分钟；同主体全局 15 次 / 10 分钟后封禁 2 小时。
- `RECOVERY_VERIFY`：3 次 / 24 小时后封禁 72 小时；同主体全局 5 次 / 24 小时后封禁 7 天。
- `MIGRATION_REQUEST`：5 次 / 10 分钟后封禁 30 分钟；同主体全局 12 次 / 10 分钟后封禁 2 小时。
- `MIGRATION_CONFIRM`：6 次 / 10 分钟后封禁 30 分钟；同主体全局 15 次 / 10 分钟后封禁 2 小时。
- `PAIRING_RESET`：3 次 / 24 小时后封禁 24 小时；同主体全局 5 次 / 24 小时后封禁 72 小时。
- 维度固定：同时统计三层键值
  - `scene + subject_hash + client_fingerprint_hash`
  - `scene + subject_hash + GLOBAL`
  - `scene + GLOBAL + client_fingerprint_hash`
- `window_start` 统一归一化规则（UTC）：
  - 10 分钟窗口场景：`(date_trunc('hour', timezone('UTC', now())) + floor(extract(minute from timezone('UTC', now())) / 10) * interval '10 minute') AT TIME ZONE 'UTC'`
  - 24 小时窗口场景：`date_trunc('day', timezone('UTC', now())) AT TIME ZONE 'UTC'`
- MVP 阶段终端全局阈值先与“同主体全局阈值”保持一致，后续按实测再单独调参。
- 本轮验证固定口径：终端全局阈值 = 同主体全局阈值；若后续单独调参，需通过新变更单独评审。
- MVP 固定策略：失败计数不在成功校验后主动清空，统一按窗口自然过期恢复。

### 7.1 打卡成对合法性触发器

> `END > START` 无法通过单行 `CHECK` 表达，使用约束触发器。

```sql
CREATE OR REPLACE FUNCTION validate_punch_pair()
RETURNS TRIGGER AS $$
DECLARE
  v_date DATE;
  v_start TIMESTAMPTZ;
  v_end TIMESTAMPTZ;
BEGIN
  IF TG_OP = 'DELETE' THEN
    RETURN OLD;
  END IF;

  -- 软删除属于合法操作，不在数据库层阻断
  IF NEW.deleted_at IS NOT NULL THEN
    RETURN NEW;
  END IF;

  -- UPDATE 改归属日期时，需同时校验新旧 local_date，避免遗留孤立 END
  FOR v_date IN
    SELECT DISTINCT d
      FROM unnest(
        ARRAY[NEW.local_date]
        || CASE
             WHEN TG_OP = 'UPDATE' AND OLD.local_date IS DISTINCT FROM NEW.local_date
             THEN ARRAY[OLD.local_date]
             ELSE ARRAY[]::DATE[]
           END
      ) AS t(d)
  LOOP
    SELECT at_utc
      INTO v_start
      FROM punch_records
     WHERE user_id = NEW.user_id
       AND local_date = v_date
       AND type = 'START'
       AND deleted_at IS NULL
     LIMIT 1;

    SELECT at_utc
      INTO v_end
      FROM punch_records
     WHERE user_id = NEW.user_id
       AND local_date = v_date
       AND type = 'END'
       AND deleted_at IS NULL
     LIMIT 1;

    IF v_end IS NOT NULL AND v_start IS NULL THEN
      RAISE EXCEPTION 'invalid punch pair: END requires START (local_date=%)', v_date;
    END IF;

    IF v_start IS NOT NULL AND v_end IS NOT NULL AND v_end <= v_start THEN
      RAISE EXCEPTION 'invalid punch pair: END must be later than START (local_date=%)', v_date;
    END IF;
  END LOOP;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE CONSTRAINT TRIGGER trg_validate_punch_pair
AFTER INSERT OR UPDATE OF at_utc, deleted_at, type, local_date
ON punch_records
DEFERRABLE INITIALLY DEFERRED
FOR EACH ROW
EXECUTE FUNCTION validate_punch_pair();
```

### 7.3 迁移状态机触发器（防止越级流转）

```sql
CREATE OR REPLACE FUNCTION enforce_migration_status_transition()
RETURNS TRIGGER AS $$
DECLARE
  v_writer_device_id UUID;
BEGIN
  IF TG_OP = 'INSERT' THEN
    -- 插入前先自动清理已过期 PENDING，防止唯一索引阻塞新请求
    UPDATE migration_requests
       SET status = 'EXPIRED',
           updated_at = now()
     WHERE user_id = NEW.user_id
       AND status = 'PENDING'
       AND expires_at <= now();

    SELECT writer_device_id
      INTO v_writer_device_id
      FROM users
     WHERE user_id = NEW.user_id
     FOR SHARE;

    IF NOT FOUND THEN
      RAISE EXCEPTION 'user not found for migration request';
    END IF;

    IF NEW.mode = 'NORMAL' AND NEW.from_device_id IS DISTINCT FROM v_writer_device_id THEN
      RAISE EXCEPTION 'invalid normal migration source: from_device_id % must match current writer %',
        NEW.from_device_id, v_writer_device_id;
    END IF;

    IF NEW.status <> 'PENDING' THEN
      RAISE EXCEPTION 'invalid initial migration status: %', NEW.status;
    END IF;
    IF NEW.expires_at <= now() THEN
      RAISE EXCEPTION 'invalid migration expiry: expires_at % must be in the future',
        NEW.expires_at;
    END IF;
    RETURN NEW;
  END IF;

  IF NEW.mode IS DISTINCT FROM OLD.mode
     OR NEW.from_device_id IS DISTINCT FROM OLD.from_device_id
     OR NEW.to_device_id IS DISTINCT FROM OLD.to_device_id
     OR NEW.recovery_code_verified IS DISTINCT FROM OLD.recovery_code_verified
     OR NEW.expires_at IS DISTINCT FROM OLD.expires_at THEN
    RAISE EXCEPTION 'migration request immutable fields cannot be changed';
  END IF;

  IF now() > NEW.expires_at AND NEW.status IN ('CONFIRMED', 'COMPLETED') THEN
    RAISE EXCEPTION 'migration request expired at %, cannot move to %',
      NEW.expires_at, NEW.status;
  END IF;

  IF NEW.status = OLD.status THEN
    RETURN NEW;
  END IF;

  IF OLD.status = 'PENDING' THEN
    IF NEW.status IN ('CONFIRMED', 'REJECTED', 'EXPIRED') THEN
      RETURN NEW;
    END IF;
    IF NEW.status = 'COMPLETED' AND NEW.mode = 'FORCED' THEN
      RETURN NEW;
    END IF;
    RAISE EXCEPTION 'invalid migration transition: % -> %', OLD.status, NEW.status;
  END IF;

  IF OLD.status = 'CONFIRMED' THEN
    IF NEW.status IN ('COMPLETED', 'EXPIRED') THEN
      RETURN NEW;
    END IF;
    RAISE EXCEPTION 'invalid migration transition: % -> %', OLD.status, NEW.status;
  END IF;

  RAISE EXCEPTION 'invalid migration transition from terminal status: %', OLD.status;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_enforce_migration_status_transition
BEFORE INSERT OR UPDATE OF status, mode, from_device_id, to_device_id, recovery_code_verified, expires_at
ON migration_requests
FOR EACH ROW
EXECUTE FUNCTION enforce_migration_status_transition();
```

```sql
CREATE OR REPLACE FUNCTION validate_web_binding_version()

      v_pairing_code_version, NEW.pairing_code_version;
```

### 7.6 限流窗口归一化触发器

```sql
CREATE OR REPLACE FUNCTION normalize_security_window_start()
RETURNS TRIGGER AS $$
DECLARE
  v_now_utc TIMESTAMP := timezone('UTC', now());
BEGIN
  IF NEW.scene IN ('WEB_PAIR_BIND', 'MIGRATION_REQUEST', 'MIGRATION_CONFIRM') THEN
    NEW.window_start :=
      (
        date_trunc('hour', v_now_utc)
        + floor(EXTRACT(MINUTE FROM v_now_utc) / 10) * interval '10 minute'
      ) AT TIME ZONE 'UTC';
  ELSIF NEW.scene IN ('RECOVERY_VERIFY', 'PAIRING_RESET') THEN
    NEW.window_start := date_trunc('day', v_now_utc) AT TIME ZONE 'UTC';
  ELSE
    RAISE EXCEPTION 'unsupported security scene for window normalization: %', NEW.scene;
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_normalize_security_window_start
BEFORE INSERT OR UPDATE OF scene, window_start
ON security_attempt_windows
FOR EACH ROW
```

### 7.9 错误标识规范（SQLSTATE + error_key）

- 业务拒绝类异常统一使用 `SQLSTATE + error_key` 双标识：
  - `SQLSTATE`：统一使用 `P0001`（业务规则拒绝）
  - `error_key`：使用 `UPPER_SNAKE_CASE`，写入错误消息前缀：`[error_key=<KEY>] ...`
- 验证记录提取规则固定：使用正则 `\\[error_key=([A-Z0-9_]+)\\]` 从错误消息提取 `error_key`；未命中即判失败。
- 当前文档中旧示例若未携带前缀，`001_init.sql` 落地时必须按本节规范补齐，不得沿用无 `error_key` 文本异常。

建议模板：

```sql
RAISE EXCEPTION USING
  ERRCODE = 'P0001',
  MESSAGE = '[error_key=MIGRATION_SOURCE_MISMATCH] invalid normal migration source';
```

建议 `error_key` 基线（与触发器规则一一对应）：

- `PUNCH_END_REQUIRES_START`
- `PUNCH_END_NOT_AFTER_START`
- `MIGRATION_SOURCE_MISMATCH`
- `MIGRATION_IMMUTABLE_FIELDS`
- `MIGRATION_TRANSITION_INVALID`
- `WEB_BINDING_REACTIVATE_DENIED`
- `SYNC_COMMIT_STALE_WRITER`
- `AUTO_PUNCH_ON_FULL_DAY_LEAVE`
- `FULL_DAY_LEAVE_WITH_AUTO_PUNCH`
- `RECORD_USER_ID_IMMUTABLE`
