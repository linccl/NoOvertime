# 拍照打卡与日志上传配置说明

## 新增接口
- `POST /api/v1/punch-photos/upload`
  - Header: `Authorization: Bearer <mobile_token>`
  - Form:
    - `punch_record_id`
    - `local_date`
    - `punch_type`
    - `file`
- `POST /api/v1/logs/upload`
  - Header: `Authorization: Bearer <mobile_token>`
  - Form:
    - `log_date`
    - `log_kind`：可选，`all` / `error` / `info`，默认 `all`
    - `file`

成功响应统一返回：

```json
{
  "url": "https://...",
  "object_key": "punch-photos/... or logs/...",
  "expires_at": "2026-05-14T10:00:00Z",
  "request_id": "..."
}
```

## 存储后端
- 默认 `local`
- 可切换为 `oss`
- 当前支持:
  - 通用上传配置 `UPLOAD_*`
  - 照片专用覆盖 `PUNCH_PHOTO_UPLOAD_*`
  - 日志专用覆盖 `LOG_UPLOAD_*`

如果未设置照片或日志专用配置，会回退复用通用 `UPLOAD_*` 配置。

## 配置项
- `UPLOAD_STORAGE_BACKEND`
  - `local` 或 `oss`
- `UPLOAD_LOCAL_DIR`
  - 本地存储目录；`local` 模式必填，默认 `./uploads-data`
- `UPLOAD_PUBLIC_BASE_URL`
  - 公开访问前缀；本地模式会拼成 `${base}/uploads/...`
- `UPLOAD_OSS_ENDPOINT`
- `UPLOAD_OSS_BUCKET`
- `UPLOAD_OSS_ACCESS_KEY_ID`
- `UPLOAD_OSS_ACCESS_KEY_SECRET`
- `UPLOAD_OSS_PREFIX`

照片专用覆盖项，与通用项同语义：

- `PUNCH_PHOTO_UPLOAD_STORAGE_BACKEND`
- `PUNCH_PHOTO_UPLOAD_LOCAL_DIR`
- `PUNCH_PHOTO_UPLOAD_PUBLIC_BASE_URL`
- `PUNCH_PHOTO_UPLOAD_OSS_ENDPOINT`
- `PUNCH_PHOTO_UPLOAD_OSS_BUCKET`
- `PUNCH_PHOTO_UPLOAD_OSS_ACCESS_KEY_ID`
- `PUNCH_PHOTO_UPLOAD_OSS_ACCESS_KEY_SECRET`
- `PUNCH_PHOTO_UPLOAD_OSS_PREFIX`

日志专用覆盖项，与通用项同语义：

- `LOG_UPLOAD_STORAGE_BACKEND`
- `LOG_UPLOAD_LOCAL_DIR`
- `LOG_UPLOAD_PUBLIC_BASE_URL`
- `LOG_UPLOAD_OSS_ENDPOINT`
- `LOG_UPLOAD_OSS_BUCKET`
- `LOG_UPLOAD_OSS_ACCESS_KEY_ID`
- `LOG_UPLOAD_OSS_ACCESS_KEY_SECRET`
- `LOG_UPLOAD_OSS_PREFIX`

照片走低频访问 OSS，日志走标准 OSS：

```bash

PUNCH_PHOTO_UPLOAD_STORAGE_BACKEND=oss
PUNCH_PHOTO_UPLOAD_OSS_ENDPOINT=https://oss-cn-beijing.aliyuncs.com
PUNCH_PHOTO_UPLOAD_OSS_BUCKET=noovertime-photo-prod
PUNCH_PHOTO_UPLOAD_OSS_ACCESS_KEY_ID=photo-ak
PUNCH_PHOTO_UPLOAD_OSS_ACCESS_KEY_SECRET=photo-sk
PUNCH_PHOTO_UPLOAD_OSS_PREFIX=noovertime/prod

LOG_UPLOAD_STORAGE_BACKEND=oss
LOG_UPLOAD_OSS_ENDPOINT=https://oss-cn-beijing.aliyuncs.com
```

线上当前推荐做法：

- 配置文件：`/root/lincc/lincc-project/NoOvertime/config.production.json`
- 启动方式：固定二进制 + `systemd`
- 服务名：`noovertime-api.service`

对应 `systemd` 示例：

```ini
[Unit]
Description=NoOvertime API
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
WorkingDirectory=/root/lincc/lincc-project/NoOvertime
Environment=CONFIG_FILE=/root/lincc/lincc-project/NoOvertime/config.production.json
ExecStart=/root/lincc/lincc-project/NoOvertime/bin/noovertime-api
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```

建议在阿里云侧将：

- `noovertime-photo-prod` Bucket 默认存储类型设置为低频访问
- `noovertime-log-prod` Bucket 默认存储类型设置为标准存储

## 对象路径约定

- 打卡照片：

```text
punch-photos/<user_id>/<yyyy-mm-dd>/<punch_record_id>.<ext>
```

- 应用日志：

```text
logs/<user_id>/<device_id>/<log_kind>/<yyyy-mm-dd>.log
```

如果配置了 `*_OSS_PREFIX`，对象会自动挂到对应前缀下，例如：

```text
noovertime/prod/punch-photos/<user_id>/<yyyy-mm-dd>/<punch_record_id>.jpg
noovertime/prod/logs/<user_id>/<device_id>/<log_kind>/<yyyy-mm-dd>.log
```

## 分桶建议

- 照片单独 Bucket，默认存储类型设为低频访问
- 日志单独 Bucket，默认存储类型设为标准存储

原因：

- 照片通常是一次上传、60 天内极少读取，适合低频访问
- 日志对象通常更小，且同一用户同一设备同一天可能重传覆盖；如果日志 Bucket 也用低频访问，30 天内替换或删除会更容易触发不足规定时长费用
- `log_kind` 拆分后，手动全量日志与自动错误日志会落到不同对象路径，不再互相覆盖

## 保留策略
- 服务端为照片与日志记录 `expires_at`
- 服务端启动后会跑后台清理循环，同时每次成功上传后也会顺带清理已过期对象
- 已过期对象删除后，对应数据库记录会标记 `deleted_at`
- 照片上传额外要求用户为会员；非会员 token 会被服务端直接拒绝
- `sync/commits` 会回填 `membership_tier` 与 `membership_expires_at`，供 Android 端决定照片同步能力
