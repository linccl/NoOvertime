# 回归门禁口径与落地清单（T301）

## 1. 目标与边界

- 目标：把 `scripts/pg18_regression.sh` 纳入固定门禁链路，基于 `99_summary.md` 做最终 pass/fail 判定，并将本地/CI 统一入口收敛为 `make gate`（回归门禁入口为 `make check`）。
- 强边界：
  - 不修改 `scripts/pg18_regression.sh` 的核心逻辑。
  - 不修改历史归档目录内任何文件。

## 2. 门禁入口命令

- 统一入口：`make gate`
- 回归门禁入口：`make check`（只执行回归门禁与 summary 判定）
- 建议别名：
  - `make regression`：执行回归门禁脚本。
  - `make test`：可复用 `make regression`，确保团队默认测试入口也会触发门禁。
- CI（若启用 GitHub Actions）也只调用 `make gate`，避免双口径。

> 注意：若未覆盖连接参数（命令行/`REGRESSION_*`/`PG*`），会直接使用 `scripts/pg18_regression.sh` 内的默认连接配置；本地运行建议显式指定，避免误连远端或被网络策略阻断。

## 3. Summary 判定规则（门禁真值）

门禁脚本读取 `scripts/pg18_regression.sh` 输出目录中的 `99_summary.md`，按以下规则判定：

- 必须以 `PASS` 开头（允许 `PASS` 或 `PASS (...)`）：
  - `PG18 Gate`
  - `First migration run`
  - `Second migration expected failure (42710)`
  - `Rollback drill expected failure (42P01)`
  - `Rollback clean check`
  - `Supplemental C09 (cross transaction)`
  - `Supplemental C10 (cross transaction)`
- 必须为 `NONE`：
  - `Non-pass IDs from suite output`

判定失败条件：

- 任一必需字段缺失或无法解析。
- 任一 `PASS` 字段值不是以 `PASS` 开头。
- `Non-pass IDs from suite output` 不是 `NONE`（包括 `SKIPPED`）。

## 4. 退出码约定（run_regression_gate.sh）

- `0`：门禁通过（回归脚本成功，且 summary 全部满足规则）。
- `1/2/3/4/5...`：`scripts/pg18_regression.sh` 原样失败码透传（任意非 `0` 均透传，常见为 `1/2/3/4/5`）。
- `6`：门禁脚本执行异常（runner 缺失/不可执行、workdir 无法解析、summary 缺失/不可读/不可解析）。
- `7`：summary 可读但判定为失败（存在非 PASS/NONE 项）。

说明：`make check` 只要收到非 `0` 即判定失败。

## 5. 本批次文件落地清单

- `Makefile`（新增）：提供 `regression`、`test`、`check`、`gate` 目标；`check` 为回归门禁入口，`gate` 为统一验收入口。
- `scripts/run_regression_gate.sh`（新增）：封装回归执行、summary 判定与退出码映射。
- `.github/workflows/regression.yml`（可选新增）：`push` 与 `pull_request` 执行 `make gate`。
- 文档文件（新增或修改，二选一）：
  - `README.md`（若项目已有/需补齐），或
  - `docs/回归门禁说明.md`（推荐）。
- `docs/沟通进展日志.md`（修改）：第 6 章末尾追加记录 201（含时间、文件列表、sha256、使用说明、影响、下一步）。

## 6. 执行顺序（依赖）

1. 先实现 `scripts/run_regression_gate.sh`。
2. 再接入 `Makefile` 的 `check` 入口。
3. 可选接入 `.github/workflows/regression.yml`。
4. 更新使用文档。
5. 最后追加 `docs/沟通进展日志.md` 记录 201。
