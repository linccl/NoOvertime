# NoOvertime 实施计划（数据库与接口联调）

> 状态：Draft v1.4  
> 日期：2026-02-07  
> 依据文档：`docs/需求文档.md`、`docs/数据库方案草案.md`（v0.7）

## 1. 背景

- 当前数据库方案已在文档层闭环（约束、触发器、迁移状态机、Web 绑定版本校验）。
- 下一步目标是把规则从“文档”变成“可执行资产”，并完成最小联调验证。

## 2. 目标

- 产出可执行 PostgreSQL 初始化迁移脚本。
- 产出可联调 API 契约草案（覆盖同步/迁移/配对码/Web 只读绑定）。
- 通过最小正反例验证，确认关键约束真实生效。

## 3. 范围

### 3.1 本期包含

- 新增初始化迁移脚本：`db/migrations/001_init.sql`
- 新增接口契约文档：`docs/API契约草案.md`
- 新增验证文档（含执行命令与结果）：`docs/数据库验证记录.md`
- 在 `docs/沟通进展日志.md` 追加实施记录

### 3.2 本期不包含

- Android/Web 客户端代码实现
- 生产部署编排与监控告警
- 历史版本数据迁移（仅初始化建库）

## 4. 复审后补齐项（相对原 Plan）

- 增加“迁移脚本可重放策略”说明（一次性脚本 + 明确不可重复执行段）。
- 增加“失败回滚策略”（事务边界、手工回滚清单）。
- 增加“迁移状态机负例验证”与“Web 撤销会话不可逆验证”。
- 增加“接口鉴权/限流验收点”映射到需求条目。
- 增加交付清单与完成定义（DoD），避免只有文档没有可执行证据。
- 增加“`sync_id` 重放幂等验证”（同请求重复提交不能产生脏写）。
- 增加“迁移不可变字段负例验证”（`mode/from_device_id/to_device_id/recovery_code_verified/expires_at` 更新应失败）。
- 增加“恢复码重置接口契约”与“配对码重置后会话版本联测”。
- 增加“限流失败次数与封禁窗口”验证，确保 `security_attempt_windows` 可支撑需求安全基线。
- 收敛“请假原始记录同步边界”：`leave_records` 纳入同步载荷。
- 补齐“限流三层维度”落地要求：细粒度、主体全局、终端全局（`scene + GLOBAL + client_fingerprint_hash`）。
- 增加“迁移脚本重放与失败回滚演练”验收点，避免只有理论策略无实测证据。

## 5. 分步计划

1. 固化迁移目录结构  
- 创建 `db/migrations/` 目录及 `001_init.sql`。

2. 落地数据库初始化脚本  
- 按顺序写入：`extension -> enum -> table -> index -> function -> trigger`。  
- 覆盖 `docs/数据库方案草案.md` v0.7 全部约束。

3. 编写 API 契约草案  
- 覆盖接口：  
  - 同步上报（Punch/LeaveRecord/DaySummary/MonthSummary 原子提交）  
  - 迁移申请/确认/强制接管  
  - 配对码重置  
  - 恢复码重置  
  - Web 只读绑定创建与鉴权  
- 对每个接口给出：请求、响应、错误码、幂等字段、鉴权要求、限流策略。  
- 固化 `sync_id` 语义：  
  - 相同 `sync_id + payload_hash`：幂等成功（不重复生效）  
  - 相同 `sync_id + 不同 payload_hash`：拒绝并返回冲突错误
- 明确 `FR-031` 映射：`FULL_DAY` 请假日出现 `source=AUTO` 打卡时，接口返回与数据库拒绝原因一致的错误码。
- 明确 `FR-032` 映射：写接口对打卡时间按分钟向下取整（截断）后入库；当 `local_date/timezone_id/minute_of_day` 与截断后时间戳不一致时返回参数错误，并与数据库拒绝原因对齐。
- 补充错误码映射附录：数据库拒绝原因与 API 错误码一一对齐（含限流阻断、幂等冲突、状态机越级、撤销会话不可逆、`stale/non-writer device commit`、封禁窗口内拒绝），并明确低版本提交按 NOOP 丢弃（不覆盖已落库高版本）。

3.1 限流阈值基线（MVP）  
- `WEB_PAIR_BIND`：5 次 / 10 分钟后封禁 30 分钟；同主体全局 15 次 / 10 分钟后封禁 2 小时。  
- `RECOVERY_VERIFY`：3 次 / 24 小时后封禁 72 小时；同主体全局 5 次 / 24 小时后封禁 7 天。  
- `MIGRATION_REQUEST`：5 次 / 10 分钟后封禁 30 分钟；同主体全局 12 次 / 10 分钟后封禁 2 小时。  
- `MIGRATION_CONFIRM`：6 次 / 10 分钟后封禁 30 分钟；同主体全局 15 次 / 10 分钟后封禁 2 小时。  
- `PAIRING_RESET`：3 次 / 24 小时后封禁 24 小时；同主体全局 5 次 / 24 小时后封禁 72 小时。  
- 维度固定为三层并行统计：`scene + subject_hash + client_fingerprint_hash`、`scene + subject_hash + GLOBAL`、`scene + GLOBAL + client_fingerprint_hash`。  
- MVP 阶段终端全局阈值先与“同主体全局阈值”保持一致，后续按实测单独调参。  
- 本轮验证固定口径：终端全局阈值 = 同主体全局阈值；若后续单独调参，需通过新变更单独评审。  
- MVP 固定策略：失败计数不在成功后主动清空，统一按窗口自然过期恢复。

4. 执行建库与语法验证  
- 在空库执行 `001_init.sql`，确认一次通过。  
- 在同库二次执行，确认“不可重复执行段”按预期失败且错误可定位。  
- 演练失败回滚清单（事务回滚 + 手工清理步骤），验证可恢复到干净状态。  
- 验证所需测试种子数据通过独立 SQL 注入，不写入 `001_init.sql`。  
- 记录执行命令与结果摘要。

5. 执行关键正反例验证  
- 正例：普通迁移、强制迁移、配对码重置撤销旧绑定、`leave_records` 与汇总原子同步、相同 `sync_id + payload_hash` 重试幂等成功、普通迁移来源设备与当前写入端一致时可通过、存在过期 `PENDING` 请求时新迁移申请可成功并自动将旧请求转为 `EXPIRED`、限流封禁窗口到期后请求可恢复通过、打卡时间含秒/毫秒的 API 请求会被截断到分钟后成功写入、限流窗口 `window_start` 在入库时按 UTC 规则被归一化、同实体高版本同步可覆盖已落库旧版本。  
- 反例：  
  - 普通迁移越级完成（应失败）  
  - 普通迁移 `from_device_id` 非当前 `writer_device_id`（应失败）  
  - 过期迁移请求流转到 `CONFIRMED/COMPLETED`（应失败）  
  - 非当前写入端或过期 `writer_epoch` 的 `sync_commits` 提交（应失败）  
  - 同步实体以低版本覆盖高版本数据（应被丢弃且不覆盖）  
  - 迁移请求关键字段更新（应失败）  
  - 迁移请求 `expires_at` 续期更新（应失败）  
  - 打卡或请假记录 `user_id` 更新改写（应失败）  
  - 全天请假日写入 `source=AUTO` 打卡（应失败）  
  - 绕过接口直写数据库时，打卡时间非分钟粒度（`at_utc` 含秒/毫秒）写入（应失败）  
  - 打卡 `local_date/timezone_id/minute_of_day` 与时间戳不一致写入（应失败）  
  - 限流请求传入伪造 `window_start` 后，入库值仍未按 UTC 规则归一化（应失败）  
  - 撤销会话恢复为 ACTIVE（应失败）  
  - `ACTIVE` Web 绑定会话 `pairing_code_version` 与当前用户版本不一致（应失败）  
  - 缺 START 直接写 END（应失败）  
  - 相同 `sync_id + 不同 payload_hash`（应失败）  
  - 处于封禁窗口期间请求仍被放行（应失败）  
  - 连续错误尝试未触发封禁窗口（应失败）  
  - 终端全局维度未触发封禁窗口（应失败）

- 在 `docs/沟通进展日志.md` 追加实施与验证结果。

## 6. 验收标准（DoD）

- `db/migrations/001_init.sql` 在空 PostgreSQL 库可执行通过。
- 关键负例被数据库明确拒绝或按既定门控策略丢弃（有可复现 SQL 与结果信息）。
- API 契约覆盖需求关键条目：`FR-008/FR-016/FR-021/FR-022/FR-023/FR-026/FR-027/FR-028/FR-029/FR-031/FR-032/FR-033`。
- `sync_id` 幂等重放验证通过：  
  - 相同 `sync_id + payload_hash` 重试为幂等成功  
  - 相同 `sync_id + 不同 payload_hash` 被拒绝
- 版本门控验证通过：低版本同步不会覆盖高版本已落库数据，且高版本同步可覆盖旧版本。
- 单写入端兜底验证通过：非当前写入端或过期 `writer_epoch` 的 `sync_commits` 提交被数据库拒绝。
- 限流封禁验证通过（按 3.1 阈值在三层维度进入阻断窗口，并在窗口自然过期后恢复）。
- 限流窗口归一化验证通过：`window_start` 入库值符合 UTC 10 分钟/24 小时边界规则。
- 重放/回滚验证通过：初始化脚本二次执行按预期失败，且可按清单恢复环境。
- 验证记录与沟通日志均已更新，能支持后续回溯。

## 7. 风险与应对

- 风险：触发器顺序或依赖错误导致建库失败。  
  - 应对：严格按依赖顺序组织 DDL，并在空库实跑。
- 风险：接口契约与数据库约束不一致。  
  - 应对：API 错误码与触发器异常一一映射。
- 风险：后续实现绕过标准入口（如直接更新状态字段）。  
  - 应对：数据库层保持不可篡改约束，服务层只走标准事务接口。
- 风险：一次性初始化脚本重复执行导致冲突。  
  - 应对：在验证文档中明确“仅用于空库初始化”，后续变更走增量迁移编号。
- 风险：限流策略只在文档定义、未形成可执行验证。  
  - 应对：在 `docs/数据库验证记录.md` 固化限流场景测试步骤与结果。
- 风险：终端全局阈值过严导致误封正常客户端。  
  - 应对：MVP 先与主体全局阈值对齐，验证记录沉淀命中分布，再迭代细化阈值。
