# NoOvertime 需求文档

> 本文档用于沉淀需求与决策，随沟通滚动更新。任何未明确事项默认视为**未决**，需在“未决问题”中闭环。

## 1. 背景

- 项目代号：NoOvertime
- 当前状态：全新项目（仓库初始化阶段）
- 核心业务问题：在弹性工作制下，消除员工对“工时差值”的不确定性（何时下班才满 9 小时、当月到底可调休还是需补班）。

## 2. 目标（Goals）

- 提供一个用于记录上下班打卡时间、自动计算工时与调休/补班的移动端 App。
- 用户在设置的时间段内启动 App 时自动打卡（上班/下班分别触发不同逻辑）。
- 将“半小时计量（超出向下取整、不足向上取整）”规则自动化，实时量化调休资产（正为可调休、负为需补班）。
- App 侧完成所有计算并展示数据；支持同步到云数据库。
- 提供 Web 端只读数据看板（无打卡功能），从云数据库读取并展示统计结果。

## 3. 范围（Scope）

### 3.1 本期包含（In Scope）

- 移动端 App
  - 自动打卡：上班时间段触发上班打卡；下班时间段触发下班打卡
  - 手动打卡：支持手动打卡、补卡、修改打卡时间
  - 统计与展示：每日工时、月总工时、月可调休时长、月需补班时长
  - 请假管理：支持登记/修改/删除（上午半天/下午半天/全天）
  - 本地存储：所有数据先落手机本地
  - 云同步：每次生成/修改/删除打卡或请假记录即自动同步；也支持手动点击“同步”
  - 日期/月份切换：查看任意日期、任意月份的统计与列表
- Web 端（只读）
  - 从云数据库读取并展示：每日工时、月总工时、月可调休时长、月需补班时长
  - 支持日期/月份切换与列表浏览（无打卡、无计算）
- 云数据库
  - 存储移动端同步的原始记录与（或）计算结果，供 Web 端展示

### 3.2 本期不包含（Out of Scope）

- Web 端打卡
- 在未启动 App 的情况下后台自动打卡（仅在“启动 App”时触发）
- 跨月结转/抵扣调休（各月份独立计算，互不影响，详见业务规则）
- Web 端扫码配对（MVP 仅输入配对码）
- iOS 客户端（MVP 仅 Android）

## 4. 用户与场景

### 4.1 目标用户

- 个人用户：追求工作生活平衡（WLB）的弹性办公开发者/白领
  - 制度环境：公司实行弹性上下班，但要求每天满固定工时（如 9 小时）
  - 心理诉求：不想多干无谓加班，也不想因计算失误导致迟到/调休不准
  - 使用习惯：手机端负责打卡与计算；电脑端（Web）用于查看统计

### 4.2 核心使用场景（Top Scenarios）

- 场景 A：上班时间段内启动 App → 自动上班打卡 → 晚些时候下班时间段内启动 App → 自动下班打卡 → 生成当日工时与调休增减
- 场景 B：当月查看任意日期工时、当月累计工时与调休/补班余额
- 场景 C：登记请假（上午/下午/全天）→ 当天不累计也不消耗调休
- 场景 D：手机端同步到云 → Web 端查看同样的统计与列表
- 场景 E：手动打卡/补卡/修改打卡时间 → 重新计算当日与当月统计 → 同步到云端供 Web 查看

## 5. 需求拆解

> 建议我们先补齐“用户—场景—价值—验收”，再进入具体功能/页面/接口设计。

### 5.1 用户故事（User Stories）

- 作为【个人用户】，我希望【自动记录上下班打卡时间】，以便【不用手动记工时】。
- 作为【个人用户】，我希望【自动计算每日工时与当月累计】，以便【快速知道是否达标】。
- 作为【个人用户】，我希望【把超出/不足 9 小时的部分按规则换算为调休/补班】，以便【清楚当月可调休或需补班时长】。
- 作为【个人用户】，我希望【手机端数据可同步到云端、Web 端只读查看】，以便【在电脑上方便浏览统计】。

### 5.2 功能需求（Functional Requirements）

- FR-001：支持记录每日上班打卡时间（1 次）与下班打卡时间（1 次）。
- FR-002：支持配置“上班时间段”“下班时间段”；用户在对应时间段内启动 App 时自动触发对应打卡逻辑。
- FR-003：默认自动打卡时间段：上班 08:00-10:00；下班 17:00-19:00（均可配置）。
- FR-004：自动打卡上班与自动打卡下班在同一天内各自只触发一次（避免重复打卡）。
- FR-005：支持手动打卡（上班/下班）。
- FR-006：支持补卡：选择日期与时间补录上班/下班打卡。
- FR-007：支持修改已存在的打卡时间（上班/下班）。
- FR-008：下班打卡限制：当日缺少上班打卡时，禁止触发下班打卡（自动与手动均适用；可先补录上班再下班）。
- FR-009：统计计算前置条件：当日必须同时存在上班与下班打卡，才计算“当日工时”及对“当月调休/补班余额”的影响。
- FR-010：支持展示并计算每日工时（基于当日上/下班打卡时间）。
- FR-011：支持计算并展示当月月总工时（当月每日工时累计）。
- FR-012：支持计算并展示当月“可调休时长/需补班时长”（同一余额的正负展示）。
- FR-013：支持请假管理（登记/修改/删除）：上午半天（09:00-12:00）/下午半天（12:00-18:00）/全天。
- FR-014：请假当天：无论工作多久，均不可累计调休时长，也不可消耗调休时长（当日调休变动为 0）。
- FR-015：支持移动端本地存储。
- FR-016：每次生成/修改/删除一条打卡或请假记录即自动同步到云数据库；同时提供手动点击“同步”。
- FR-017：Web 端仅从云数据库读取并展示统计结果（无打卡、无计算）。
- FR-018：支持日期切换与月份切换查看数据。
- FR-019：提供“日期列表”视图：仅展示选中月份的数据（默认当月）。
- FR-020：提供“月份列表”视图：仅展示选中年份的 12 个月（默认当年）。
- FR-021：支持移动端生成“配对码”，用于 Web 端绑定只读查看。
- FR-022：支持重置配对码（重置后旧配对码失效）。
- FR-023：支持换机迁移：新手机输入配对码发起迁移请求，旧手机确认后新手机接管写入，旧手机强制退出。
- FR-024：配对码默认掩码显示；点按可显示明文。
- FR-025：配对码支持一键复制。
- FR-026：支持生成“恢复码”（用于旧机不可用时的强制迁移/接管），默认掩码显示；点按可显示明文；支持一键复制。
- FR-027：恢复码仅展示一次（首次生成或重置后）；若用户未保存，需要重置生成新恢复码（旧恢复码失效）。
- FR-028：支持强制迁移：旧机丢失/损坏/无法联网确认时，新手机使用“配对码 + 恢复码”接管写入；旧机后续上线需强制退出。
- FR-029：支持删除打卡记录（上班/下班）；删除后需重算当日与当月统计并同步。
- FR-030：当同一天已存在同类型打卡记录时，再次打卡/补卡需弹窗二次确认；确认后按“修改”覆盖原记录。
- FR-031：全天请假当日禁用自动打卡（仅允许手动打卡/补卡/修改/删除）。
- FR-032：打卡时间以“分钟”为最小粒度存储与计算（秒/毫秒必须为 0，即分钟精度）。
- FR-033：移动端需生成并同步“月度汇总（MonthSummary）”，Web 端直接读取展示月总工时与月可调休/需补班（不在 Web 端做计算）。

### 5.3 非功能需求（Non-functional Requirements）

- 性能：关键交互（首页加载、日期切换、月份切换、日期列表刷新、月份列表刷新）需在 2 秒内完成响应（不考虑网络延迟）。
- 可用性：离线可用（本地记录与计算）；联网后自动同步，支持手动同步重试。
- 安全：云端访问需鉴权（MVP 不引入独立账号体系，采用配对码绑定 Web 只读查看 + 恢复码迁移）；配对码/恢复码/迁移相关接口需限流与失败次数限制以避免暴力猜测；避免在日志/同步数据中泄露敏感信息。
- 兼容性：移动端仅 Android；Web 端桌面浏览器优先，移动端浏览器自适应非必需（可选）。

## 6. 业务规则与数据口径

### 6.1 时间口径与基础定义

- 标准日工时：9 小时（= 540 分钟）。
- 正常上班时间：09:00-18:00（用于“迟到”判断与默认配置参考，不限制弹性打卡）。
- 迟到规则：上班打卡时间 **晚于 09:10** 标记为迟到（09:10 含当作未迟到）；仅在“非请假日且非周末（周六/周日）”时判定。
- 弹性工作制：只要当日实际工时达到 9 小时即可视为达标，不限制具体上下班时间。
- 午休/休息扣除：不扣除（工时按下班-上班计算）。
- 时区与日期：以打卡当时手机的本地时区确定 `localDate`；出差跨时区按当地时区记录与统计。
- 时间精度：打卡时间以分钟为最小粒度存储与计算；秒/毫秒必须为 0（分钟精度）。

### 6.2 每日工时计算

- 当日工时（分钟）= 下班打卡时间 - 上班打卡时间。
- 当日没有成对打卡（缺上班或缺下班）时：不计算当日工时；当日也不参与当月工时与调休/补班累计。

### 6.3 调休/补班计算（非请假日）

> 计量单位：30 分钟（半小时）。

- 设 `workMinutes` 为当日工时（分钟），`requiredMinutes = 540`。
- 差值：`delta = workMinutes - requiredMinutes`
  - 若 `delta > 0`（超出）：当日调休变动 = `floor(delta / 30) * 30`（分钟）
  - 若 `delta < 0`（不足）：当日调休变动 = `-ceil(abs(delta) / 30) * 30`（分钟）
  - 若 `delta = 0`：当日调休变动 = 0

示例（与需求口径一致）：

- 09:11 上班，18:31 下班：超出 20 分钟 → 不累计调休
- 09:11 上班，18:41 下班：超出 30 分钟 → +30 分钟调休
- 09:11 上班，19:01 下班：超出 50 分钟 → +30 分钟调休
- 09:11 上班，19:11 下班：超出 60 分钟 → +60 分钟调休
- 09:11 上班，18:01 下班：少 10 分钟 → -30 分钟调休
- 09:11 上班，17:41 下班：少 30 分钟 → -30 分钟调休
- 09:11 上班，17:31 下班：少 40 分钟 → -60 分钟调休

### 6.4 请假规则

- 请假类型：
  - 上午半天：09:00-12:00
  - 下午半天：12:00-18:00
  - 全天
- 请假当天：无论当日工作多久，均**不累计**调休，也**不扣减**调休（当日调休变动固定为 0）。
- 请假当天：若当日上下班成对打卡，仍可展示“当日实际工时”（下班-上班）；但不影响当月调休/补班余额。

### 6.5 月度统计与展示规则

- 月总工时：当月每日工时累加（仅累加可计算的 `workMinutes`；未成对打卡的不计入；展示见 6.9）。
- 当月调休余额：按“选中月份”范围内的每日“调休变动”累加，各月份独立计算；不与其他月份结转/抵扣。
- 请假日：`adjustMinutes` 固定为 0；月总工时若可计算则照常计入。
- 统计日期口径：不区分工作日/周末/节假日；只要当日可计算工时（上下班成对）就纳入当月统计。
- 展示口径：
  - 当月调休余额 ≥ 0：展示为“月可调休时长 = 余额”
  - 当月调休余额 < 0：展示为“月需补班时长 = abs(余额)”

### 6.6 自动打卡规则（App 启动触发）

- 自动打卡触发条件：App 进入前台（含冷启动、从后台回前台）时，若当前时间落在配置的时间段内则触发。
- 全天请假：若当日请假类型为“全天”，禁用自动打卡（不生成 `START/END`）；但允许手动打卡/补卡/修改/删除。
- 上班自动打卡：
  - 当前时间在“上班时间段”内，且当日尚无上班打卡 → 生成一条 `START` 打卡（`source=AUTO`）。
  - 当日已存在上班打卡 → 不再重复生成。
- 下班自动打卡：
  - 当前时间在“下班时间段”内，且当日已存在上班打卡、但尚无下班打卡 → 生成一条 `END` 打卡（`source=AUTO`）。
  - 当日缺少上班打卡 → 禁止生成下班打卡（与 FR-008 一致）。
  - 当日已存在下班打卡 → 不再重复生成。
- 配置校验：上班时间段与下班时间段不得重叠；若配置重叠，必须阻止保存并提示修正。

### 6.7 手动打卡 / 补卡 / 修改

- 手动打卡：在任意时间可手动生成 `START` 或 `END` 打卡（受“缺上班禁止下班”约束）。
- 补卡：选择日期与时间补录 `START`/`END`（受“每天各 1 次”约束；若已存在同类型打卡，需弹窗确认后按“修改”覆盖）。
- 修改：允许修改已存在的上班/下班打卡时间；修改后需触发重算并同步。
- 删除：允许删除已存在的上班/下班打卡记录；删除后需触发重算并同步。
- 唯一性约束：同一 `localDate` 下，`START` 最多 1 条、`END` 最多 1 条（重复写入按“修改”或提示处理）。
- 冲突处理：当同一天已存在同类型打卡记录时，再次手动打卡/补卡必须弹窗二次确认；确认后按“修改”覆盖原记录。
- 合法性校验：
  - 若同时存在 `START` 与 `END`，要求 `END > START`，否则视为无效并阻止保存。
- 跨日限制：不允许跨日打卡；同一 `localDate` 的 `START/END` 需都落在该 `localDate` 内，且 `END` 必须晚于 `START`。

### 6.8 同步与一致性口径

- 移动端为计算与写入的唯一端；Web 端只读展示（不修改）。
- 同步触发：
  - 每次生成/修改/删除一条打卡或请假记录后，立即尝试同步到云端。
  - 提供手动“同步”用于重试失败或补发。
- 同步内容：
  - 原始打卡记录（PunchRecord）
  - 请假原始记录（LeaveRecord）
  - 每日汇总（DaySummary）：在当日上下班成对后生成/更新；请假日也可生成汇总（`isLeaveDay=true`，`adjustMinutes=0`）
  - 月度汇总（MonthSummary）：由移动端按月重算并同步（供 Web 端读取展示月总工时与月可调休/需补班）
- 统计口径与同步时机：
  - 当日上下班未成对时，不生成可计算的 `workMinutes/adjustMinutes`；移动端与 Web 端显示“未完成（缺上班/缺下班）”，并仅展示已存在的打卡时间。
  - 当日上下班成对且非请假日 → 计算并更新 `workMinutes/adjustMinutes`，并更新当月统计展示。
  - 当日上下班成对且为请假日 → 计算并更新 `workMinutes`（用于展示），`adjustMinutes` 固定为 0。
- Web 端鉴权（MVP）：Web 端通过输入配对码绑定只读查看权限；配对码可在移动端重置，重置后旧配对码失效。
- 配对码规格（MVP）：8 位数字；长期有效（直到手动重置），无过期机制。
- 多设备（MVP）：仅允许 1 台手机端作为写入端；Web 端只读查看。
- 换机迁移（MVP）：新手机通过配对码发起迁移请求；旧手机需确认；确认后旧手机强制退出且不可继续写入，新手机成为唯一写入端。
- 强制迁移（MVP）：若旧手机丢失/损坏/无法联网确认，新手机使用“配对码 + 恢复码”强制接管写入；接管后旧手机若再次上线需强制退出且不可继续写入。
- 恢复码（MVP）：16 位字母数字混合（A-Z、0-9，不区分大小写）；首次生成或重置后仅展示一次；云端仅保存哈希（不保存明文）；若遗失需重置生成新码（旧码失效）。

### 6.9 时长展示口径

- 调休/补班：以 0.5 小时步进展示（30 分钟 = 0.5 小时）。
- 工时：以“X 小时 Y 分钟”展示（Y 为 0-59）。
- 月总工时：同工时口径展示（小时+分钟）。

## 7. 约束与假设（Constraints & Assumptions）

- 一切计算在移动端 App 内完成；Web 端仅展示云端数据，不参与计算。
- 数据以移动端本地存储为主；同步到云数据库用于跨端查看与备份。
- MVP 账号体系：移动端生成配对码，Web 端输入配对码获得只读查看权限（不做扫码）。
- MVP 平台范围：仅 Android。
- 云数据库：`PostgreSQL 18.1`（Docker 部署）。
- 鉴权方式（MVP）：不引入独立账号体系；Web 端通过配对码绑定只读查看，强制迁移通过“配对码 + 恢复码”完成接管。

## 8. 验收标准（Acceptance Criteria）

- [ ] 自动打卡：在上班时间段内启动 App 会生成上班打卡；在下班时间段内启动 App 会生成下班打卡（且同一天不重复生成）。
- [ ] 全天请假禁用自动打卡：当日请假类型为“全天”时，App 进入前台不会生成自动打卡记录。
- [ ] 迟到判断：仅在非请假日且非周末（周六/周日）时，上班打卡晚于 09:10 标记迟到；请假日与周末不判迟到（周末仍统计工时，若上下班成对）。
- [ ] 工时达标：当日上下班成对且实际工时为 9 小时（=540 分钟）时，调休变动=0。
- [ ] 调休累计向下取整：超出 9 小时的部分按 30 分钟向下取整累计（覆盖文档示例）。
- [ ] 调休扣减向上取整：不足 9 小时的部分按 30 分钟向上取整扣减（覆盖文档示例）。
- [ ] 请假日不增不减：请假当天调休变动始终为 0。
- [ ] 请假日工时展示：请假当天若上下班成对打卡，可展示“当日实际工时”，但不影响当月调休/补班余额。
- [ ] 月度展示：当月余额为正/0 显示可调休；为负显示需补班。
- [ ] 同步后可在 Web 端查看：Web 端展示与手机端一致的统计结果（以最后一次同步为准）。
- [ ] 缺上班禁止下班：当日无上班打卡时，无法生成下班打卡（自动/手动/补卡均适用）。
- [ ] 禁止跨日：同一 `localDate` 的上下班打卡必须落在同一天且 `END > START`，否则无法保存或不参与统计。
- [ ] 逐条同步：每次生成/修改/删除一条打卡或请假记录，都会触发一次同步（失败可手动重试）。
- [ ] 补卡/修改可重算：补卡或修改打卡时间后，当日与当月统计会更新，并可同步到 Web 端展示。
- [ ] 删除可重算：删除打卡记录后，当日与当月统计会更新，并可同步到 Web 端展示。
- [ ] 请假管理可同步：新增/修改/删除请假记录后，会触发同步，并正确更新当日与当月展示口径。
- [ ] 冲突需确认：同一天已存在同类型打卡记录时，再次打卡/补卡会弹窗确认；确认后按“修改”覆盖原记录。
- [ ] 配对码只读查看：Web 端输入配对码后可查看云端数据；重置配对码后旧码不可再访问。
- [ ] 配对码规格：配对码为 8 位数字；除非手动重置，否则不会过期。
- [ ] 自动打卡触发：App 冷启动或从后台回前台进入前台时，会按时间段规则触发自动打卡。
- [ ] 换机迁移：新手机发起迁移请求后，旧手机确认；确认后旧手机强制退出，新手机成为唯一写入端并可继续记录。
- [ ] 强制迁移：旧机不可用时，新手机使用“配对码 + 恢复码”可接管写入；旧机后续上线会被强制退出。
- [ ] 配对码交互：默认掩码显示；点按显示明文；支持一键复制。
- [ ] 恢复码交互：默认掩码显示；点按显示明文；支持一键复制；首次生成/重置后仅展示一次。
- [ ] 展示口径：调休/补班按 0.5 小时步进展示；工时按“小时+分钟”展示。
- [ ] 分钟精度：打卡时间存储与计算均按分钟（秒/毫秒必须为 0）。
- [ ] Web 端无计算：Web 端展示的月总工时与月可调休/需补班来自移动端同步的 MonthSummary，与移动端一致。
