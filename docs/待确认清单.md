# NoOvertime：需求待确认清单（给 human 拍板用）

更新时间：2026-03-14
目的：把“需要你做选择/定口径”的点一次性列清楚。其余不需要你确认的文案优化/字段对齐，我们会直接改进并落到 `docs/需求文档.md`。

> 使用方式：请你按条目编号逐个给结论（选 A/B 或写一句你想要的规则即可）。  
> 每条后面都写了“推荐默认”，你也可以直接同意推荐项。

## 已按 token-only 基线收敛

- 移动端受保护接口统一为 `Authorization: Bearer <token>`。
- `POST /api/v1/tokens/issue` / `POST /api/v1/tokens/rotate` 已落地。
- `POST /api/v1/sync/commits` 顶层不再要求客户端上传 `user_id/device_id/writer_epoch`，首次成功同步会回填 `user_id`。
- 除 `tokens/*` 与 `sync/commits` 外，匿名 token 调用当前仍有效的移动端受保护接口统一返回 `409 USER_ID_NOT_READY`。
- Web 月/日汇总查询已收敛为 `Authorization: Bearer <mobile_token>`。
- `pairing-code/*`、`recovery-code/*`、`web/read-bindings*`、`migrations/takeover`、`migrations/forced-takeover` 当前阶段均为 paused/historical：保留路由但统一返回 `410 FEATURE_PAUSED`，不再是当前有效主流程。

## 已过时口径（阅读旧文档时请忽略）

- 客户端显式上传 `X-User-ID` / `X-Device-ID` / `X-Writer-Epoch` 作为移动端写接口必需头。
- `sync/commits` 请求体顶层必须携带 `user_id/device_id/writer_epoch`。
- `migrations/requests` 必须由客户端显式上传 `user_id/from_device_id`。
- `migrations/confirm` 必须由客户端显式上传 `operator_device_id`。
- 历史口径曾把 `migrations/takeover` 视为当前有效接管路径，并继续作为 token-only 主流程推进。
- 历史口径曾把 `migrations/forced-takeover` 视为唯一有效接管路径，且必须由客户端显式上传 `to_device_id`。

## 1) 半天请假（AM/PM）算不算“请假日”？会不会判迟到？

现在文档写的是：**只有“非请假日且非周末”才判迟到**。问题在于：半天请假到底算不算“请假日”。

- 选项 1A（推荐，最省事）：**只要当天有请假记录（AM/PM/FULL_DAY 任意一种），当天就不判迟到。**
- 选项 1B：**只要“上午请假/全天请假”才不判迟到；下午请假（PM）仍按 09:10 规则判迟到。**

你拍板：`1A / 1B / 其他规则：__________`

## 2) 半天请假（AM/PM）要不要影响“自动打卡”？

现在文档明确的是：**全天请假禁用自动打卡**。半天请假要不要也“挡住一半自动打卡”，需要定。

- 选项 2A（推荐，KISS）：**只有 FULL_DAY 才禁用自动打卡；AM/PM 不影响自动打卡。**
- 选项 2B（更符合直觉）：**AM 请假禁用自动上班自动打卡（START）；PM 请假禁用自动下班自动打卡（END）；FULL_DAY 禁用全部 AUTO。**

你拍板：`2A / 2B / 其他规则：__________`

## 3) 自动打卡时间段能不能跨午夜（比如 22:00-02:00）？

跨午夜会带来“算哪一天”“localDate 怎么归属”等复杂问题。

- 选项 3A（推荐）：**不允许跨午夜**（要求 `start < end`，且同一天内）。跨午夜配置直接不让保存。
- 选项 3B：允许跨午夜（需要定义跨日归属口径，并补更多校验/验收）。

你拍板：`3A / 3B`

---

## 4) 出差跨时区：自动打卡时间段按“当前所在地时间”还是“设置时的时区”？

举例：你在上海设置了 08:00-10:00，上班出差到伦敦后，还想不想按伦敦当地 08:00-10:00 触发？

- 选项 4A（推荐，符合直觉）：**按当前手机所在地时区解释时间段**（到哪儿就按哪儿的 08:00-10:00）。
- 选项 4B：**固定按“设置时的时区”解释**（不随出差变化）。

你拍板：`4A / 4B`

---

## 5) 出差跨时区：同一次上下班可能落在不同 localDate，怎么算“成对打卡”？

如果上班在时区 A、下班在时区 B，可能出现：
- START 的 `localDate=2026-03-04`
- END 的 `localDate=2026-03-05`

这时还算不算“同一天成对”？

- 选项 5A（推荐，简单可验收）：**不支持跨时区导致的跨日成对**。出现这种情况就提示用户手动调整/补卡到同一天。
- 选项 5B：支持跨时区成对（需要定义“以哪个时区/哪一天为准”，并调整服务端/DB 约束或引入特殊逻辑）。

你拍板：`5A / 5B`

## 6) Web 只读看板：要不要展示“未完成日”（缺上班/缺下班）？

现在需求里既强调“Web 不计算”，又希望 Web 能展示缺上班/缺下班、以及已存在的打卡时间。这里必须选一种落地口径，否则实现会分叉。

- 选项 6A（推荐）：**Web 只读只看 DaySummary/MonthSummary；移动端必须在“未成对”时也同步 DaySummary（status=INCOMPLETE + start/end）。**  
  好处：Web 不需要读 PunchRecord，也不会出现“Web 侧推导规则跟手机不一致”。
- 选项 6B：**Web 允许读取 PunchRecord 做“展示级推导”（仍不做统计计算）。**  
  风险：规则容易漂移，需要把推导规则写死并做更多验收。

你拍板：`6A / 6B`

- 选项 7A（推荐）：允许（只读，不限制并发）。
- 选项 7B：限制并发（例如最多 N 个会话/设备；超出就踢掉旧的或拒绝新的）。

## 8) MonthSummary 的“月份主键”到底用 `yearMonth` 还是 `monthStart`？

目前 API/DB 草案与 T001 都是 `month_start=YYYY-MM-01`。需求文档里是 `yearMonth=YYYY-MM`。

- 选项 8A（推荐，和现有 API/DB 对齐）：**云端统一用 `monthStart=YYYY-MM-01` 作为主键/查询口径；`yearMonth=YYYY-MM` 只是 UI 展示派生字段。**
- 选项 8B：坚持 `yearMonth=YYYY-MM` 作为主键（需要改 API/DB/实现，成本更高）。

你拍板：`8A / 8B`

## 9) 配对码/恢复码到底是谁生成？

需求文档写“移动端生成”，但现 API/DB 草案更偏“服务端生成并返回”。

- 选项 9A（推荐，安全和一致性更强）：**服务端生成并返回；客户端只负责展示/复制/重置触发。**
- 选项 9B：客户端生成并上报（需要重新审视安全/冲突/重置版本等规则）。

你拍板：`9A / 9B`

## 10) 普通迁移（换机迁移）到底谁来“发起”？

需求文档描述：新手机输入配对码发起迁移，旧手机确认。  
但当前 API/DB 草案的迁移请求更像是：**由当前写入端（旧手机）发起**（因为需要 `from_device_id` 且受 writer 限制）。

- 选项 10A：坚持“新机凭配对码发起”（需要新增/调整接口与校验）。
- 选项 10B（推荐，贴近现有 API/DB）：由**旧机/当前 writer 发起迁移**，新机提供必要信息后等待确认。
- 选项 10C：两步握手：新机先“表达迁移意图”，旧机确认后才正式创建迁移请求（建议配合有效期）。

你拍板：`10A / 10B / 10C`

## 11) 迁移请求要不要有“有效期/可取消”？

前端需要明确：新机点了迁移以后，等多久算超时？能不能取消？

- 选项 11A（推荐）：有有效期（例如 10 分钟），可取消。
- 选项 11B：无有效期（一直挂起直到确认/拒绝）。

你拍板：`11A（有效期：____ 分钟） / 11B`

## 12) 强制迁移/接管后，“旧机强制退出”到底是什么意思？

- 选项 12A（推荐）：旧机**仅禁止继续写入/同步**，并给出提示（数据仍保留在旧机本地）。
- 选项 12B：旧机直接“清空本地数据/回到初始页”（更激进，风险更大）。

你拍板：`12A / 12B`

## 13) DeviceAuth（device_access_token）MVP 做不做？

API 草案里出现了 `Authorization: Bearer <device_access_token>`，但需求文档没定义 token 怎么来、怎么换、怎么失效。

- 选项 13A（推荐）：**MVP 不做真实 token 鉴权**（保留配对码只读 + writer_epoch/device_id 门控），并在文档里明确“token 不启用/不校验”或删掉相关表述。
- 选项 13B：MVP 做最小 token：需要补齐“签发/轮换/注销/泄露处理”的需求与验收。

你拍板：`13A / 13B`

## 14) 首次使用：`user_id / device_id / writer_epoch` 怎么初始化？

系统里很多接口/校验依赖这三样，但需求文档目前没写“第一次打开 App 怎么拿到它们”。

- 选项 14A（推荐）：提供一个“初始化/注册设备”流程（首次启动向服务端创建 user/device，并把自己设为 writer）。
- 选项 14B：先纯本地记录，等第一次同步时再自动在服务端创建（需要明确接口行为）。

你拍板：`14A / 14B`

## 15) 同一天同类型重复打卡/请假（不同 id）服务端怎么处理？

例如同一天出现两条 START（不同 id）。DB 目前更偏“唯一约束拒绝”。

- 选项 15A（推荐）：**强约束：同日同类型记录 id 必须稳定**，修改/补卡=同 id + version++ 覆盖；不同 id 视为非法并拒绝。
- 选项 15B：允许“替换/覆盖”（需要定义：旧记录如何处理、是否软删、版本如何递增、如何保证幂等）。

你拍板：`15A / 15B`

## 16) 软删除后允许“复活”吗？（deletedAt 置回 null）

- 选项 16A（推荐）：不允许复活。删除就是删除；想恢复只能“新版本覆盖写入”（明确规则）。
- 选项 16B：允许复活（需要定义 version 与幂等语义，复杂度更高）。

你拍板：`16A / 16B`

## 17) 恢复码“只展示一次”，要不要加一个“我已保存”确认？

- 选项 17A（推荐）：需要。用户点“我已保存”/勾选后才允许离开页面（减少丢码概率）。
- 选项 17B：不需要。只提示一次就行。

你拍板：`17A / 17B`

## 18) 自动打卡要不要一个“总开关”（一键停用）？

现在只有“配置时间段”，没有“全局关闭自动打卡”的按钮。

- 选项 18A（推荐）：提供总开关（关了就完全不自动打卡，只能手动/补卡）。
- 选项 18B：不提供（只能通过把时间段设得很窄/很离谱来变相关闭）。

你拍板：`18A / 18B`

## 19) 工程策略：默认质量门是否要包含 `make regression`？

当前团队共识是：默认 gate 跑 `fmt/vet/unit-test/build`，`make regression` 依赖外部 PG18/可能超时，按 DB 变更专项任务再加。

- 选项 19A（推荐）：默认不跑 regression；仅 DB 相关任务/CI 夜间跑。
- 选项 19B：所有任务都默认跑 regression（会更慢，且需要稳定的外部依赖）。

你拍板：`19A / 19B`
