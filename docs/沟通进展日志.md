## 3. 最终有效规则与验收门槛（当前执行口径）

- 规则优先级固定：
  - `docs/需求文档.md` -> `docs/数据库方案草案.md` -> `docs/实施计划.md` -> `docs/三项交付物执行计划.md (Frozen v1.64)`。
  - `docs/沟通进展日志.md` 仅作审计留痕，不可作为规则来源。
- 前置环境门槛：
  - `SHOW server_version_num` 必须 `>=180000 且 <190000`（当前实跑 `180001`，即 `PostgreSQL 18.1`）。
  - `plpgsql` 必须存在；`pgcrypto` 必须“已安装或可创建”，失败需保留 `SQLSTATE`。
  - 非系统 schema 白名单固定为 `public + 允许扩展的 extnamespace`。
- 迁移门槛：
  - `db/migrations/001_init.sql` 必须 `BEGIN ... COMMIT` 单事务。
  - 执行命令固定 `psql -X -v ON_ERROR_STOP=1`。
  - 任一 DDL 失败即整体回滚，不允许半成功对象残留。
- 对象完整性门槛：
  - 采用“数据库方案快照 + `001_init.sql` 清单”双基线交叉核对，核心对象域（A ∪ B）差异即失败。
  - 未命名约束统一规范化为 `UNNAMED_CONSTRAINT::<table_name>::<source_anchor>`。
- 门控与失败证据门槛：
  - API 必须输出 `gate_result(APPLIED/NOOP/REJECTED)` 与 `gate_reason`。
  - `NOOP` 门控副作用矩阵固定：`REPLAY_NOOP / SYNC_ID_CONFLICT / LOW_OR_EQUAL_VERSION`。
  - `LOW_OR_EQUAL_VERSION`（新 `sync_id`）证据固定：4 张业务表 `INSERT/UPDATE/DELETE=0` 且 `sync_commits INSERT=1, UPDATE=0, DELETE=0`。
  - 写入生效门控固定：`APPLIED_WRITE -> APPLIED`。
  - 失败标识分层固定：`P0001 -> SQLSTATE + error_key`，`23502 -> SQLSTATE + table_name + column_name`，其余约束拒绝 -> `SQLSTATE + constraint_name`。
- 审计与留痕门槛：
  - 执行前后 `sha256` 必须留痕（首次创建可记 `ABSENT`）。
  - `docs/数据库验证记录.md` 最终 hash 必须外置回写本日志，禁止自引用闭环。
  - 时间证据必须包含“动作完成时间 + 记录时间”，差值 `<=5 分钟`。
  - 复审闭环必须包含 `review_checklist_id` 与 `evidence_anchor_set (>=3)`，连续轮次不得重复。
  - 自 `v1.58` 起，连续复审轮次动作完成时间间隔需 `>=120s`，否则该轮不计入 `x/3`。
  - 自 `v1.60` 起，`evidence_anchor_set` 需要满足可复核规则（含 `anchor_id/sha256_12` 可反推）。
- 范围与结论边界：
